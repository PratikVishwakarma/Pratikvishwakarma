the problem

Platform engineers type the same commands hundreds of times a day.

Before ADB Helper: every deploy cycle meant opening a terminal, remembering the exact device IP, typing adb connect 192.168.x.x:5555, then adb root, then adb remount, then adb push, then adb shell am force-stop… repeated across multiple devices, multiple times per day. A missed step meant a broken deploy. No history. No shortcuts. Pure repetition.
ADB Helper solves it by wrapping the entire workflow in a browser UI served locally via Flask. Device connection, command execution, APK deployment, signing, log capture — all from one page. The active device is auto-injected into every adb -s call. Commands are categorised, searchable, and favouritable. Custom commands can be saved. The full history of 200+ executed commands is always one click away.
feature walkthrough

Every screen, explained.

Four screenshots covering the main workflows — device management, command execution, package management, and APK signing.

localhost:5000 · Device
screen 01
Device Commands
The most-used panel. Root device, remount system partition, reboot variants (normal / bootloader / fastboot), disable verity, factory reset — all with one click. The Connect → root → remount chain is the full deploy prep in a single action.
adb root adb remount fastboot disable-verity factory reset
KEY DETAIL
Device IP is shown in the sidebar once connected. All commands auto-inject -s <device> — no manual targeting needed.
localhost:5000 · Raw Executor
screen 02
Raw Executor
Type any ADB command freely. The active device -s flag is automatically prepended to leading adb commands — no need to remember the device serial or IP. Execute, Copy output, or Save as a custom command directly from this view.
free-form input auto -s inject execute copy save command
KEY DETAIL
Commands run here are added to History automatically. Useful for one-off getprop, dumpsys, or custom shell queries.
localhost:5000 · Install / Pkg
screen 03
Install / Package Management
Full APK lifecycle management. Install (standard or force with --no-incremental -r), uninstall, clear app data via pm clear, get APK path on device, open App Info screen, and enable/disable packages — all without touching the terminal.
install APK force install pm clear pm path enable / disable pkg App Info
STB CONTEXT
Force install with --no-incremental is essential on STB — incremental installs fail on restricted system partitions. This is baked in by default.
localhost:5000 · Sign APK
screen 04
APK Signing
System apps on AOSP require platform signing. This panel covers all signing paths: standard JKS via apksigner, platform key signing via signapk.jar with platform.pk8, signature verification with --print-certs, and the full STB sign workflow.
apksigner (JKS) platform.pk8 signapk.jar verify signature STB sign tool
WHY THIS MATTERS
AOSP system apps must be signed with the platform key. An unsigned or user-signed build silently fails to gain privileged permissions. This panel eliminates that entire class of debug confusion.

how it was built

A real example of AI-assisted tooling.

This tool wasn't designed upfront. It grew iteratively from a problem — and AI assistance was used deliberately at each stage to move faster without compromising the engineering decisions.

1
Identified the repetition problem
Noticed I was typing the same 8–10 adb commands in sequence every deploy cycle. Decided to solve it properly rather than just alias them in .zshrc.
problem definition
2
Designed the command taxonomy
Grouped all commands I use into logical categories: Device, Deploy, Install, Launch, Broadcast, Info, Logs, Sign, File Ops. This structure became the sidebar. Pure engineering decision — no AI involved here.
architecture decision
3
Flask server + ADB subprocess layer
Used AI to scaffold the Python Flask backend — route structure, subprocess execution of ADB commands, device auto-injection logic, and the history persistence layer. Reviewed and modified every piece before running it.
⚡ ai-assisted
4
Frontend UI — dark theme, sidebar, command panels
AI generated the initial HTML/CSS structure matching the dark terminal aesthetic I wanted. Iterated on layout, search behaviour, favourites logic, and the Raw Executor UX through back-and-forth prompting.
⚡ ai-assisted
5
STB-specific commands and signing flows
All platform-specific commands — the deploy chain, platform.pk8 signing, STB sign tool integration, disable-verity sequence — came from direct AOSP/STB knowledge. This is where the tool became actually useful rather than generic.
platform expertise
6
Daily use, iteration, expansion
The 200-entry history count is real usage. Commands were added as new needs appeared. My Commands section holds 7 saved one-liners specific to the STB platform — none of which exist in any public ADB reference.
production use
reflections

What this tool actually demonstrates.

platform depth
The command set reveals real AOSP knowledge — disable-verity before remount, platform.pk8 for system app signing, the exact sequence for a clean deploy. These aren't things you learn from tutorials.
ai as a multiplier
The scaffolding that would've taken a week (Flask routes, frontend boilerplate, JS state) was done in hours. The remaining time went into the parts that actually required expertise — the command taxonomy and STB-specific flows.
tooling instinct
Building tools for your own workflow is a senior engineering trait. It means you understand your problem well enough to abstract it — and you value your time enough to invest in fixing friction.
internal vs public
Not every tool needs to be open-sourced. This one has platform-specific internals that stay internal. The showcase is the engineering thinking behind it — not the code.
