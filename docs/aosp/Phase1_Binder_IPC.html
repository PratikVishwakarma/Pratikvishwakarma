Chapter 01

Binder IPC

The kernel-level IPC backbone of Android

WHY Binder Exists

Every app on Android runs in its own isolated Linux process. They cannot touch each other's memory — that is the security model. But they constantly need to communicate: STB settings app needs AudioManager, TvInputManager, WindowManager — all living in system_server, a completely different process.

Standard Linux IPC mechanisms were simply not good enough:

Linux IPC OptionWhy It Fails for Android
Shared memoryNo security — any process can map it. No ownership model.
Unix socketsYou'd design protocol, serialisation, threading, and security from scratch every time.
PipesOne-way, no object passing, no identity.
D-BusToo slow for Android's IPC volume. No kernel-level security.
🍽️ The Restaurant Analogy
Everything is a restaurant
  • Your App = Customer sitting at a table
  • Binder = The Waiter — carries orders and food between table and kitchen
  • system_server = The Kitchen — does all the actual work
  • AIDL = The printed Order Form / Menu
  • Parcel = The Tray carrying food and orders
  • ServiceManager = The Reception Desk — tells you which kitchen handles what

You (the customer) never walk into the kitchen. You never touch the stove. You tell the waiter what you want. The waiter carries your order. The kitchen prepares it. The waiter brings it back. That is Binder.

How Binder Works — The Three Layers

┌────────────────────────────┐
│   Java / Kotlin Layer      │  ← IBinder, AIDL-generated stubs
├────────────────────────────┤
│   JNI + libbinder (C++)    │  ← BpBinder, BBinder, Parcel, IPCThreadState
├────────────────────────────┤
│   Binder Kernel Driver     │  ← /dev/binder, mmap, binder_transaction
└────────────────────────────┘

The Two Participants

NameWhat It Is
BpBinderClient side — Binder Proxy. Holds a handle integer pointing to the remote object.
BBinderServer side — Binder Native. Has onTransact() called by the kernel on each transaction.
StubAIDL-generated Java wrapper around BBinder — you extend this for your service.
Stub.ProxyAIDL-generated Java wrapper around BpBinder — clients get this automatically.

A Transaction Step by Step

When your STB settings app calls TvInputManager.getTvInputList():

Step 1 — Client writes to Parcel
  AIDL Proxy.getTvInputList() packs arguments into a Parcel byte buffer

Step 2 — Kernel takes over
  BpBinder → IPCThreadState → ioctl(/dev/binder, BINDER_WRITE_READ)
  Kernel: binder_ioctl → binder_thread_write → binder_transaction()

Step 3 — ONE copy via mmap  (the efficiency secret)
  system_server has mmap'd /dev/binder
  Kernel maps client Parcel DIRECTLY into server address space — 1 copy only
  Traditional IPC = 2 copies: client→kernel buffer, kernel→server

Step 4 — Server Binder thread wakes up
  Kernel selects thread from pool (max 15 threads)
  IPCThreadState.executeCommand() → BBinder.onTransact() → your method runs

Step 5 — Reply travels back
  Result packed into reply Parcel, mapped back to client
  Client's transact() call returns ✅

ServiceManager — The Reception Desk

ServiceManager is a name → IBinder lookup table living at Binder handle 0 — hardcoded in the kernel, always known. Every service registers at boot. Clients look up by name once, then communicate directly forever after.

# Boot time — services register:
ServiceManager.addService("wifi",     wifiServiceImpl)
ServiceManager.addService("tv_input", tvInputManagerService)
ServiceManager.addService("bluetooth",bluetoothManagerService)

# App startup — look up once (one call to handle 0):
val binder = ServiceManager.getService("wifi")
val wifiManager = IWifiManager.Stub.asInterface(binder)

# Every subsequent call — direct, ServiceManager not involved:
wifiManager.connectionInfo  ← goes directly to WifiServiceImpl via stored handle

Key Concepts

ConceptWhat It Means
onewayFire-and-forget. FLAG_ONEWAY to kernel. No reply Parcel. Caller returns immediately. Used for callbacks and event notifications.
1 MB limitEntire Binder buffer per process is shared ~1 MB. Exceeding it = TransactionTooLargeException. Unpredictable because it depends on all concurrent transactions.
DeathRecipientCallback fired when remote process dies. Always use this to detect crashes and reconnect cleanly.
RemoteCallbackListThread-safe list of remote callbacks. Auto-removes dead clients. Always prefer over a plain List.
UID/PID identityKernel stamps every transaction with caller UID/PID. Cannot be spoofed. Foundation of all Android permission checks.
⚠️ STB Warning
Never pass large data through Binder

List<EpgEntry> with 5000 entries ≈ 1 MB → TransactionTooLargeException. Send a ParcelFileDescriptor (locker key) instead. Data stays in shared memory (ashmem/MemoryFile). The fd handle crossing Binder is just 4 bytes.

⚠️ STB Warning
Never make Binder calls on the main thread

Binder calls are synchronous and can block if system_server is busy (channel change, OTA, EPG load). Always call from Dispatchers.IO. Post results back to main thread via Handler or withContext(Dispatchers.Main).

Binder is a kernel driver at /dev/binder that lets isolated processes call remote methods safely — kernel verifies identity, single-copy mmap makes it efficient, ServiceManager at handle 0 is the lookup table for all named services.
Chapter 02

AIDL

Android Interface Definition Language — the contract layer

WHY AIDL Exists

Binder is the wire. Without AIDL you would manually write Parcel pack/unpack code on both sides for every method — error-prone, unreadable, and breaks the moment you add a parameter. AIDL auto-generates all that boilerplate from a simple interface definition.

🍽️ The Order Form Analogy
AIDL is the restaurant's official printed order form

Without it: customers shout random things, waiters get confused. With a printed form: exactly what can be ordered (methods), what information you must provide (parameters), what you get back (return type), and a number for each item (transaction code).

  • Stub = Kitchen's copy of the form — receives and dispatches orders
  • Stub.Proxy = Customer's copy — fills in the form and hands to waiter
  • Interface = The agreement both sides signed — neither can change unilaterally

What AIDL Generates

# Your .aidl file:
interface ITunerService {
    int tune(int channelId);
    ChannelInfo getCurrentChannel();
}

# AIDL generates ONE Java file with three things inside:

ITunerService
  ├── Stub  (server side — you extend this)
  │     ├── onTransact()   ← receives raw Parcel, dispatches by transaction code
  │     └── asInterface()  ← converts raw IBinder to typed proxy
  │
  └── Stub.Proxy  (client side — AIDL gives you this)
        ├── tune()              ← packs args, calls transact(), unpacks reply
        └── getCurrentChannel() ← same pattern

Directional Parameters — in / out / inout

🍽️ Food Direction Analogy
  • in → You send ingredients TO the kitchen. Kitchen does not send them back.
  • out → You send an empty plate. Kitchen fills it and returns it.
  • inout → You send a half-cooked dish, kitchen finishes it, returns the completed dish.

90% of the time you use in. You are asking the kitchen to do something with what you gave them.

KeywordMeaning
inData flows client → server only. Most common. Use for sending requests.
outServer fills empty object, returns to client. Less common.
inoutBoth directions. Most expensive — two Parcel writes. Use rarely.
primitivesint, long, boolean — always by value. No direction keyword needed.

oneway — Drop the Chit and Walk Away

🍽️ The Chit Box Analogy

Normal order: You sit and wait for food to arrive at your table.

oneway order: You drop a chit in the box at the counter and walk away. No food comes back to you. Kitchen processes it when free.

Used for notifications — kitchen shouting at you: "HDMI connected!", "Remote battery low!" — kitchen does not wait for your reaction.

⚠️ The danger: If you drop 100 chits too fast, the chit box overflows. Extra chits are thrown away silently. No exception, no log, just gone.

# In your .aidl file:
oneway interface IStbDataCallback {
    void onWifiStatusChanged(in WifiStatusInfo info);
    void onHdmiStateChanged(boolean connected);
    void onRemoteBatteryChanged(int percent);
}

# Generated proxy uses FLAG_ONEWAY:
mRemote.transact(TRANSACTION_onWifiStatusChanged, data, null, IBinder.FLAG_ONEWAY)
//                                                       ^^^^  ^^^^^^^^^^^^^^^^^^^
//                                              no reply Parcel   kernel returns immediately

AIDL Versioning — Critical for STB OTA

Your STB has millions of deployed devices. StbSettings updates but CC hasn't updated yet. Transaction codes are assigned by position — reordering them means old clients silently call the wrong method.

# SAFE ✅ — always add at the END:
interface IStbBrokerService {
    WifiStatusInfo      getWifiStatus();       ← v1, code=1, NEVER touch
    BluetoothStatusInfo getBluetoothStatus();  ← v1, code=2, NEVER touch
    RemoteStatusInfo    getRemoteStatus();     ← v2, added at end ✅
    NetworkInfo         getDetailedNetwork();  ← v3, added at end ✅
}

# DANGEROUS ❌ — inserting in the middle:
interface IStbBrokerService {
    WifiStatusInfo      getWifiStatus();       ← code=1 ✅
    RemoteStatusInfo    getRemoteStatus();     ← INSERTED HERE — was getBluetooth!
    BluetoothStatusInfo getBluetoothStatus();  ← code=3 now — old clients call WRONG method
}
✅ Best Practice
Always expose getVersion()

Add int getVersion(); as the first method in every AIDL interface. Return a constant you bump with every change. Clients check version before calling newer methods — graceful degradation on older STBs.

AIDL is the restaurant's official order form — it defines what can be ordered, auto-generates all packing/unpacking code on both sides, and the transaction code (item number) must never change position or old clients silently order the wrong thing.
Chapter 03

Parcel

The serialisation wire format — how data travels on the tray

WHY Parcel Exists

Binder carries data as raw bytes. Something must pack your method arguments into those bytes and unpack them on the other side. Java Serialization does this but is 10× slower (uses reflection), larger (writes metadata), and was not designed for IPC. Parcel is a flat byte buffer — raw values only, no field names, no class names, no reflection.

🍽️ The Tray Analogy
Parcel is the tray with a strict packing rule

Items must be placed left to right in a specific order. The receiver must pick them up in the exact same order. The tray has no labels — position is the only identity.

  • Pack: Channel=42, Name="HBO", Volume=80 → [42]["HBO"][80]
  • Unpack correct: readInt()=42 ✅ readString()="HBO" ✅ readInt()=80 ✅
  • Unpack wrong order: readString() tries to read int bytes as string → gibberish 💥

What Parcel Actually Is

A raw byte buffer with a read/write cursor — like a cassette tape. You write forward, you read forward. No random access.

# Writing — cursor moves forward:
val parcel = Parcel.obtain()    ← borrow from pool, NEVER new Parcel()
parcel.writeInt(42)             ← cursor: 0 → 4 bytes
parcel.writeString("HBO")       ← cursor: 4 → 16 (4 len + chars UTF-16)
parcel.writeInt(80)             ← cursor: 16 → 20

# Reading — rewind first:
parcel.setDataPosition(0)
val channel = parcel.readInt()      ← reads 42 ✅
val name    = parcel.readString()   ← reads "HBO" ✅
val volume  = parcel.readInt()      ← reads 80 ✅

parcel.recycle()  ← ALWAYS return to pool

Parcelable — Teaching Your Object to Pack Itself

# Easy way — @Parcelize generates everything:
@Parcelize
data class ChannelInfo(
    val id: Int,
    val name: String,
    val frequency: Long,
    val isHD: Boolean
) : Parcelable

# What it generates under the hood — writeToParcel order must MATCH readFromParcel:
override fun writeToParcel(parcel: Parcel, flags: Int) {
    parcel.writeInt(id)                       ← 4 bytes
    parcel.writeString(name)                  ← 4 + n*2 bytes (UTF-16)
    parcel.writeLong(frequency)               ← 8 bytes
    parcel.writeInt(if (isHD) 1 else 0)     ← bool = int in Parcel
}

# CREATOR — how AIDL rebuilds your object on the receiving side:
override fun createFromParcel(parcel: Parcel) = ChannelInfo(
    id        = parcel.readInt(),
    name      = parcel.readString() ?: "",
    frequency = parcel.readLong(),
    isHD      = parcel.readInt() != 0
)

The 1 MB Limit

system_server's Binder buffer (shared ~1 MB total):
┌──────────────────────────────────────────────────┐
│ Transaction A  (CC → wifi status)       [2 KB]  │
│ Transaction B  (Settings → display)     [4 KB]  │
│ Transaction C  (your app → EPG data)  [800 KB]  │ ← taking most of it
│ Transaction D  (another app → ...)     [???]    │ ← NO SPACE 💥
└──────────────────────────────────────────────────┘

# TransactionTooLargeException is unpredictable — depends on
# what OTHER apps are doing at the same moment

The Right Way — File Descriptor for Large Data

🍽️ The Locker Key Analogy

Instead of putting the food itself on the tray, put a locker key on the tray. The receiver uses the key to get the food directly from the locker.

  • WRONG: Tray carries 800 KB of EPG data → tray overloaded 💥
  • RIGHT: Tray carries FileDescriptor (4 bytes) → shared memory holds 800 KB → zero-copy ✅
# Sender (StbSettings):
fun getEpgDataFd(): ParcelFileDescriptor {
    val memFile = MemoryFile("epg_data", epgBytes.size)
    memFile.writeBytes(epgBytes, 0, 0, epgBytes.size)
    return memFile.fileDescriptor  ← just the key — 4 bytes on tray
}

# Receiver (CC):
val pfd = brokerService.epgDataFd
val epgBytes = FileInputStream(pfd.fileDescriptor).readBytes()  ← from shared memory
MethodBehaviour
writeTypedList(list)Uses Parcelable directly. Fast, direct byte writes. Always prefer this.
writeList(list)Uses Java Serialization internally per item. Slow, reflection. Avoid.
obtain() / recycle()Always borrow from pool, always return. Never new Parcel().
Parcel is a flat byte tray with a moving cursor — pack and unpack in exact same order, raw bytes only, 10× faster than Serialization. Never put large data directly on the tray; use a file descriptor (locker key) instead. Always obtain() from pool and recycle() when done.
Chapter 04

Zygote & Process Model

How every Android app process is born

WHY Zygote Exists

Every app runs in its own Linux process. Creating a process from scratch means loading the JVM, loading 2000+ Android framework classes, initialising the GC — this takes seconds on a slow STB SoC. That is unacceptable for app launch time.

The key insight: most of this work is identical for every app. String.class, ArrayList.class, Context.class — the same bytes, the same memory, loaded fresh every time. What if you did it once and just copied the result?

👔 The Manager + Google Doc Analogy
Zygote is a fully equipped master desk

The manager creates a fully equipped master office desk: computer ready, all standard software installed, all tools loaded. Instead of setting up a new desk from scratch for every new employee — share a Google Doc link (fork). Each employee gets an instant perfect copy. They just add their own files on top.

  • Master Desk = Zygote process (JVM + 2000 framework classes pre-loaded)
  • Employee = Your app process (CC, StbSettings, any app)
  • Google Doc link = fork() — instant, zero cost
  • Employee edits = app writing its own objects — only then is a private copy made

Copy-on-Write — Why the Fork is Free

👔 The Google Doc CoW Analogy

Manager shares a 50-page Google Doc with 5 employees. No printing happens — everyone just gets a link. Employee 1 edits page 3 only → Google makes a private copy of just page 3 for Employee 1. Pages 1,2,4-50 are still shared with everyone. A copy is only made at the moment someone writes, and only for the exact page being written.

Zygote memory (50 MB of framework):
  String.class ◄──── CC app  ← shared, read-only, zero extra RAM
  ArrayList    ◄──── CC app  ← shared, read-only
  ART runtime  ◄──── CC app  ← shared, read-only

When CC creates its own objects:
  CC heap      ← only these pages get a private copy  (new RAM used)
  CC APK code  ← only this is new

# 5 apps running = still ~50 MB for framework, NOT 5 × 50 MB = 250 MB

Boot Flow — Power-ON to Your App

Power ON
  └─► Kernel boots
         └─► init (PID 1) reads /init.rc
                ├─► servicemanager   ← Binder reception desk, handle 0
                ├─► surfaceflinger   ← display compositor
                └─► Zygote (app_process)
                          │
                          │  loads ART + 2000+ classes, opens socket, WAITS
                          │
                          └─► fork() → system_server
                                           │
                                     starts all services
                                     registers with ServiceManager
                                           │
                                     Device READY  ← ACTION_BOOT_COMPLETED
                                           │
                                     User opens CC
                                           │
                          AMS sends request via Unix socket → Zygote
                                           │
                          Zygote fork() → CC process born ✅
⚙️ Why Unix Socket, Not Binder?
The Chicken-and-Egg Problem

Binder needs system_server to work properly. But system_server is spawned by Zygote. So Zygote cannot use Binder to receive spawn requests — Binder is not ready yet. Unix sockets exist at the kernel level from the very start, no dependencies on anything above it.

Cold / Warm / Hot Start

Start TypeWhat HappensTypical Time (STB)
Cold StartProcess doesn't exist. AMS → Zygote fork() → new process → load APK → onCreate()400ms – 2s
Warm StartProcess alive, Activity was destroyed. No fork. New Activity instance → onCreate()100 – 400ms
Hot StartProcess + Activity both in memory. Just onResume().<100ms
✅ STB Overlay Strategy

CC overlay should always be a hot start. If the STB OOM killer kills CC's process under memory pressure, the next open is a cold start and users notice the delay. Keep CC's process warm with a persistent background service or by ensuring your StbBrokerService connection keeps it alive.

Zygote is a pre-loaded master process that eliminates repetitive JVM + framework initialisation — every app is an instant fork() inheriting all framework classes via Copy-on-Write shared memory, only paying for its own APK code and heap.
Chapter 05

system_server Boot Sequence

How 80+ Android services start in the correct order

WHY Boot Order Matters

Ever seen ServiceNotFoundException: tv_input not found on your STB at startup? Or features that work after a few seconds but fail immediately after boot? Services start in a strict order. Calling a service before it registers means it simply does not exist yet.

🍽️ Restaurant Opening Analogy
You can't serve customers before the kitchen is ready
  • 6:00 AM — Security unlocks building (kernel + init)
  • 6:30 AM — Kitchen infrastructure ON (servicemanager, surfaceflinger)
  • 7:00 AM — Head chef sets up master kitchen (Zygote loads)
  • 7:30 AM — Sous chefs arrive, prep stations (Bootstrap: AMS, PackageManager)
  • 8:00 AM — Junior chefs arrive (Core + Other: WiFi, BT, TvInput...)
  • 9:00 AM — OPEN FOR CUSTOMERS 🎉 (ACTION_BOOT_COMPLETED)

The Three Phases Inside SystemServer.run()

AOSP source: frameworks/base/services/java/com/android/server/SystemServer.java

SystemServer.run()
  │
  ├── Phase 1: startBootstrapServices()   ← critical, everything depends on these
  │
  ├── Phase 2: startCoreServices()         ← important but not critical path
  │
  └── Phase 3: startOtherServices()        ← everything else (60+ services)

Phase 1 — Bootstrap Services

ServiceWhy First
WatchdogVery first. Monitors system_server. Hang for 60s = kill + device reboot.
InstallerTalks to installd daemon. Needed before PackageManager can read packages.
ActivityManagerServiceMost important. Manages all app lifecycles, processes, and tasks.
PowerManagerServiceBefore display, before anything visual.
PackageManagerServiceScans all installed APKs. Slowest on first boot after OTA.

Phase 3 — Other Services

startOtherServices():
  WindowManagerService      ← windows, z-order, input routing
  InputManagerService       ← remote control, keyboard, touch
  WifiService               ← what getSystemService("wifi") returns
  BluetoothManagerService
  AudioService
  DisplayManagerService
  TvInputManagerService     ← your STB's most-used service
  ... 60+ more ...

  # LAST — signal everything is ready:
  mActivityManagerService.systemReady(() -> {
      startSystemUi(context)   ← launches status bar, launcher
  })

PHASE_* Constants — Fine-Grained Ordering

Phase ConstantWhat It Means
PHASE_WAIT_FOR_DEFAULT_DISPLAY = 100Display initialised — services needing screen can proceed
PHASE_SYSTEM_SERVICES_READY = 500Core services done — getSystemService() works reliably from here
PHASE_DEVICE_SPECIFIC_SERVICES_READY = 520OEM/vendor services ready — on STB: tuner, CAS, HDMI ready HERE
PHASE_THIRD_PARTY_APPS_CAN_START = 600All system services done — apps with BOOT_COMPLETED can start
PHASE_BOOT_COMPLETED = 1000Home screen visible — ACTION_BOOT_COMPLETED broadcast — fully ready
🔴 Critical — Watchdog
Binder Deadlock = Device Reboot

Watchdog pings AMS, WMS, and other critical services every 60 seconds. If any service does not respond — Watchdog kills system_server and init restarts the device. A Binder deadlock in your custom system service can hang system_server → trigger Watchdog → unexpected device reboot. Always avoid circular Binder calls between services.

system_server starts services in three ordered phases via PHASE_* constants — Watchdog runs first and monitors everything — your device is not truly ready until PHASE_BOOT_COMPLETED = 1000 and ACTION_BOOT_COMPLETED is broadcast.
Chapter 06

HwBinder & HAL

The framework-to-hardware bridge — Project Treble

WHY HAL Exists

Before Android 8, every manufacturer (Samsung, Xiaomi, Jio) modified core Android OS to make their hardware work. Every Android update required every vendor to manually merge their changes — taking 1–2 years. Devices never got updates.

🔌 The Power Socket Analogy
A universal plug and adapter system

Every country has different power sockets. But your phone charger works everywhere because it has a standard plug + adapter.

  • Android OS = The appliance (same plug shape everywhere)
  • HAL Interface = The standard plug shape Google defines
  • Vendor HAL impl = The adapter Jio/Broadcom makes for their socket
  • Hardware chip = The wall socket (different per device)

Android OS never changes its plug shape. Vendors only make a new adapter when their hardware changes. Both sides update independently.

Project Treble — The Hard Line

BEFORE Treble (Android 7 and below):
┌─────────────────────────────────┐
│   Android OS + Vendor code      │  ← all mixed together
│ (framework + drivers + HAL)     │     update one = rebuild everything
└─────────────────────────────────┘

AFTER Treble (Android 8+):
┌─────────────────────────────────┐
│         Android OS              │  ← system partition — Google updates this
│    (pure AOSP framework)        │
├─────────────────────────────────┤  ← THE LINE (vendor interface / HAL)
│         Vendor code             │  ← vendor partition — Jio/Broadcom owns this
│   (HAL implementations)         │
│   (kernel drivers, firmware)    │
└─────────────────────────────────┘

Three Binder Domains

Binder DeviceWho Uses ItServiceManager
/dev/binderApps ↔ system_server (framework domain)servicemanager
/dev/hwbindersystem_server ↔ HAL processes (hardware domain)hwservicemanager
/dev/vndbinderHAL process ↔ HAL process (vendor domain)vndservicemanager

  └─► /dev/binder
         └─► TvInputManagerService (system_server)

                       └─► TvInputService (your tuner app process)

                                                          └─► RF Signal → channel 📺

HIDL vs AIDL HAL

GenerationDetails
HIDL (Android 8-10)Hardware Interface Definition Language. Invented just for HAL. Different syntax and tooling. Still on many deployed STBs.

STB HAL Landscape

Standard Android HALs (Google defined, vendor implemented):
  android.hardware.tv.tuner      ← DVB-S/C/T tuner chip
  android.hardware.tv.hdmi.cec   ← HDMI CEC control
  android.hardware.audio         ← audio routing, Dolby
  android.hardware.bluetooth     ← BT chip
  android.hardware.wifi          ← WiFi chip

  com.jio.hardware.cas           ← Conditional Access System
  com.jio.hardware.smartcard     ← CAM/smartcard interface
  com.jio.hardware.remotectl     ← RF remote control
HAL is a stable contract — Android OS presses standard buttons, vendor hardware responds in its own way — HwBinder (/dev/hwbinder) is the dedicated wire between them — @VintfStability freezes the contract so system and vendor partitions can update independently.
Chapter 07

Looper / Handler

The thread message-loop engine powering all of Android's event processing

WHY Looper Exists

A regular thread runs top to bottom and dies. Android needs threads that stay alive and keep processing work as it arrives. A naive while(true) loop burns CPU constantly. Looper solves this with event-driven sleeping — zero CPU when idle.

📮 The Post Office Analogy
A worker who stays at the desk all day

Without Looper: Worker arrives → delivers one letter → goes home. Next letter arrives = nobody there 😱

With Looper: Worker arrives → sits at desk all day → processes each letter → waits for next

  • Looper = The post office desk — keeps thread alive, runs the loop
  • MessageQueue = The letter tray — ordered queue of pending messages
  • Message = The letter — contains what work to do + when to do it
  • Handler = The person who sends letters AND processes them when received

How They Connect

Thread A (any thread — sender)        Thread B (has Looper — receiver)
──────────────────────────────        ────────────────────────────────
                                      Looper.prepare()
                                      val handler = Handler(Looper.myLooper()!!)
                                      Looper.loop()  ← BLOCKS HERE, waiting

handler.post {                ──►     MessageQueue receives runnable
    updateUI()                         Looper wakes up
}                                      Runnable executes on Thread B
                                       Looper goes back to waiting...

handler.postDelayed({         ──►     Inserted with target = now + 2000ms
    hideSpinner()                      Looper sleeps until exactly that time
}, 2000)                               wakes up, executes, sleeps again

The Main Thread Already Has a Looper

Your app's main thread has a Looper set up by Android before your code runs. Every single thing in your app's UI thread goes through this one loop:

# ActivityThread.java — AOSP source — this starts your app:
public static void main(String[] args) {
    Looper.prepareMainLooper()   ← sets up main thread Looper
    ActivityThread thread = new ActivityThread();
    thread.attach(false, startSeq);
    Looper.loop()                ← your app runs 'inside' this loop FOREVER
}

# Everything is a message posted to this loop:
Activity.onCreate()        → posted as a message
View.invalidate()          → posted as a message
runOnUiThread { }          → posted as a message
Touch / key events         → posted as a message
Binder callbacks           → posted as a message  ← CRITICAL for STB

The Critical Binder + Looper Connection

🔴 Most Common STB Mistake
Binder callbacks arrive on a Binder thread — NEVER on main thread

When a remote process calls your AIDL callback, it arrives on one of Binder's thread pool threads. Touching any UI directly = CalledFromWrongThreadException.

# WRONG:
override fun onWifiStatusChanged(info: WifiStatusInfo) {
    textView.text = info.ssid  ← 💥 on Binder thread, NOT main thread
}

# RIGHT:
override fun onWifiStatusChanged(info: WifiStatusInfo) {
    mainHandler.post { textView.text = info.ssid }  ← ✅
    // or with coroutines:
    mainScope.launch { _wifiState.value = info }    ← ✅
}

STB Production Patterns

# Pattern 1 — Signal strength polling every second:
private val pollRunnable = object : Runnable {
    override fun run() {
        updateSignalUI(tunerService.signalStrength)
        handler.postDelayed(this, 1000)  ← reschedule itself
    }
}
fun start() = handler.post(pollRunnable)
fun stop()  = handler.removeCallbacks(pollRunnable)

# Pattern 2 — Debounce remote control channel up/down:
private val tuneRunnable = Runnable { tunerService.tune(pendingChannelId) }

fun onChannelUpPressed() {
    handler.removeCallbacks(tuneRunnable)    ← cancel previous
    handler.postDelayed(tuneRunnable, 300)  ← wait 300ms for key repeat to stop
}

Choreographer — Looper Driving Every Frame

⚙️ How Your STB Renders at 60fps

Choreographer sits on top of the main Looper and fires on every hardware VSYNC signal (every 16ms at 60fps). It posts a message to the main thread. The Looper picks it up and runs: input callbacks → animation callbacks → measure/layout/draw → sleep until next VSYNC.

At 60fps you have exactly 16ms per frame. If any Handler message on the main thread takes longer than 16ms — Choreographer misses its slot → frame dropped → jank on your STB UI.

ConceptWhat It Means
Looper.loop()Blocking call that never returns. Sleeps via epoll when queue empty — zero CPU burn.
HandlerThreadPre-built background thread with Looper. Always call quitSafely() when done.
ChoreographerSits on main Looper. Drives 60fps rendering via VSYNC. 16ms budget per frame.
IdleHandlerRuns when MessageQueue is empty. Good for non-urgent prefetching on STB.
postDelayed()Inserts message with target time. Looper sleeps until then. No polling, no busy-wait.
Looper keeps a thread alive by blocking on a MessageQueue — Handler posts messages from any thread — Looper wakes, dispatches one message at a time, sleeps again — Binder callbacks always arrive on a Binder thread, so always post to main thread via Handler before touching UI.

Interview Q&A

Top questions with production-level answers — click to expand
What is Binder and why does Android use it instead of standard Linux IPC?+
Binder is a custom kernel driver (/dev/binder) designed for Android's security and performance requirements. Standard Linux IPC lacks kernel-enforced caller identity, requires 2 data copies, and has no built-in thread management. Binder verifies UID/PID in the kernel (no spoofing), does a single copy via mmap, and manages Binder thread pools automatically.
What is the Binder transaction size limit and how do you work around it?+
~1 MB per process, shared across all concurrent transactions. Exceeding it = TransactionTooLargeException. Fix: pass large data via ParcelFileDescriptor backed by MemoryFile (ashmem). The fd handle is 4 bytes on the tray; data stays in shared memory — zero-copy access from both processes.
What is the difference between a regular Binder call and oneway?+
Regular: synchronous, caller blocks until server returns, return values allowed. oneway: fire-and-forget via FLAG_ONEWAY, caller returns immediately, no return value, goes to async queue on server. Risk: oneway queue can overflow and silently drop calls with no exception thrown.
How does ServiceManager work?+
ServiceManager is the Binder context object at handle 0 — hardcoded in the kernel, always known. All named services register at boot with addService(name, binder). Clients call getService(name) once to get an IBinder handle, then communicate directly without going through ServiceManager again.
What does AIDL generate and why is it useful?+
AIDL generates a Java interface with two inner classes: Stub (server — extends Binder, has onTransact() dispatch table, you subclass this) and Stub.Proxy (client — wraps IBinder, packs/unpacks Parcel for each method). Without AIDL you would write this serialisation boilerplate manually on both sides for every method.
Why does Android use Parcel instead of Java Serialization?+
Parcel writes raw byte values only — no field names, no class names, no reflection. This makes it ~10x faster and much smaller than Serializable. It is designed for a single IPC trip, not persistence. Order of read must exactly match order of write — that is why Parcelable must implement both writeToParcel and CREATOR.createFromParcel in matching order.
What is Zygote and why does it exist?+
Zygote is a pre-loaded master process containing the JVM and ~2000 Android framework classes. Every app is a fork() of Zygote — inheriting all framework classes via Copy-on-Write shared memory at zero RAM and time cost. Without Zygote, every app launch would reload the entire framework from scratch — taking seconds on a slow STB SoC.
What is Copy-on-Write in the context of Zygote?+
CoW is a Linux kernel technique where forked processes share the parent's physical memory pages as read-only. A private copy of a page is only made when either process actually writes to it. Zygote exploits this so 10 apps can share 50 MB of framework — actual RAM usage stays ~50 MB total, not 10 × 50 MB = 500 MB.
Why does AMS use a Unix socket to talk to Zygote instead of Binder?+
Chicken-and-egg: Binder needs system_server to work properly, but system_server is spawned by Zygote. Binder is not ready when Zygote starts. Unix sockets exist at the kernel level from the very start — no dependency on anything above the kernel layer.
What is HAL and why does Project Treble matter?+
HAL (Hardware Abstraction Layer) is a stable contract between Android OS and vendor hardware code. Google defines standard interfaces (@VintfStability). Vendors implement them for their specific hardware. Project Treble (Android 8) drew a hard line between system and vendor partitions — enabling Google to update Android OS and vendors to update HALs completely independently, eliminating the 1-2 year update lag.
What is the difference between /dev/binder, /dev/hwbinder, and /dev/vndbinder?+
/dev/binder: apps ↔ system_server (framework domain). /dev/hwbinder: system_server ↔ HAL processes (hardware domain, managed by hwservicemanager). /dev/vndbinder: vendor process ↔ vendor process (vendor domain). Three separate security domains — vendor code cannot Binder-call into app space directly.
What thread do Binder callbacks arrive on?+
Binder callbacks always arrive on one of Binder's thread pool threads — never the main thread. Touching any UI directly from a Binder callback = CalledFromWrongThreadException. Always post UI work back to main thread via mainHandler.post{} or mainScope.launch{}.
What is Looper.loop() and why does it never return?+
Looper.loop() is a blocking infinite loop that pulls messages from MessageQueue and dispatches them to their target Handler. It sleeps via epoll when the queue is empty — zero CPU burn. It never returns because the thread's entire purpose is to keep processing messages. The main thread's entire lifetime is inside this loop, started in ActivityThread.main().
