Chapter 01

The Three-Layer Sandwich

Why Android can't talk to hardware directly — and the architecture that solves it

WHY This Architecture Exists

Your Kotlin code cannot talk to hardware directly. That's a Linux kernel problem. The kernel doesn't speak Java. And even if it did — you don't want 50 different apps all poking hardware registers directly. That's chaos, security nightmares, and zero coordination.

Android built a deliberate three-layer sandwich to solve this. Every hardware interaction on your STB goes through all three layers, every single time.

Your App / System Service
Java/Kotlin — the world you live in. Policy, permissions, business logic.
JAVA / KOTLIN
HAL (Hardware Abstraction Layer)
C/C++ — speaks both Java and kernel. Translator and gatekeeper. Runs in vendor process.
C / C++
Linux Kernel Driver
Speaks only hardware — ALSA, DRM/KMS, V4L2, evdev. Never knows about Android.
KERNEL SPACE
Hardware
DAC, display PHY, HDMI controller, IR receiver, video decoder
SILICON
🏢 The Corporate Hierarchy Analogy
You never talk to the factory floor directly
  • You (developer) = office employee — you request things through proper channels
  • System Service = department manager — coordinates requests, enforces rules
  • HAL = factory supervisor — knows both the office language and the machine language
  • Kernel driver = the machine operator — speaks only machine, not office

Nobody skips layers. The office employee never walks onto the factory floor. The machine operator never attends board meetings. Layers exist because each party speaks a different language and has different responsibilities.

Real Path on STB — HDMI Detection

Your Kotlin code
    getSystemService(Context.HDMI_CONTROL_SERVICE)
            │
            ▼  ← Binder IPC — crosses process boundary
    HdmiControlService  (system_server)
            │
            ▼  ← Binder IPC — crosses another process boundary
    HDMI HAL  (vendor process — Amlogic/Broadcom implementation)
            │
            ▼  ← ioctl() system call — crosses kernel boundary
    Kernel HDMI driver  (/dev/hdmi or similar)
            │
            ▼
    HDMI PHY hardware
📺 STB Context
Why you never see vendor HAL source code

The Amlogic or Broadcom HAL implementation that runs on your STB hardware is proprietary binary code. The vendor team ships it as a compiled .so file. HAL is deliberately a legal firewall — Android framework is open source, vendor implementation is not. You consume it through the AIDL HAL interface without ever seeing the source. This is by design.

5 Anchor Statements · Three-Layer Architecture
01Every hardware interaction crosses three layers — app/service → HAL → kernel. No shortcuts. No layer skipping.
02HAL is both an abstraction layer and a legal firewall — framework is open source, vendor HAL implementation is proprietary binary.
03Each layer boundary is a process boundary on modern Android — app process, system_server, HAL vendor process, kernel.
04The kernel only receives ioctl() system calls from the HAL — it never knows anything about Android, Java, or Binder.
05On STB, HDMI, audio codec, IR receiver, video decoder all follow this exact path — app → system_server → vendor HAL process → kernel driver.
Android can't let apps talk to hardware directly — that's chaos. The three-layer sandwich creates clean separation: policy in Java, translation in C/C++ HAL, execution in the kernel. Every hardware call on your STB crosses all three layers.
Chapter 02

System Services

system_server, the service registry, and the full boot-to-call path

WHY System Services Exist

Your app, the launcher, the OTT player — all separate processes. But they all need access to the same shared resources: the display, the audio hardware, the network stack, input devices.

The solution: One process (system_server) owns all shared resources. Apps request access via Binder IPC. system_server enforces permissions and coordinates access. This is why WindowManagerService, AudioService, ActivityManagerService all live together in system_server — not for convenience, but because only one process should own each piece of shared state.

⚠️ Critical Distinction — What Lives Where
Services inside system_server vs separate processes

AudioManager, HdmiControlManager, WindowManager, ActivityManager, PackageManager — all live inside system_server. One process, shared Binder thread pool.

AudioFlinger, SurfaceFlinger — these are separate processes with their own independent Binder thread pools. Not in system_server.

Boot Lifecycle — How Services Are Born

init process → Zygote → system_server

Inside SystemServer.java:
private void startBootstrapServices() {
    // First phase — most critical services
    ActivityManagerService ams = ActivityManagerService.Lifecycle.startService(...);
    ServiceManager.addService("activity", ams);

    WindowManagerService wms = WindowManagerService.main(...);
    ServiceManager.addService("window", wms);
}

private void startOtherServices() {
    // Later phase — everything else
    AudioService audioService = new AudioService(context);
    ServiceManager.addService(Context.AUDIO_SERVICE, audioService);

    HdmiControlService hdmi = new HdmiControlService(context);
    ServiceManager.addService(Context.HDMI_CONTROL_SERVICE, hdmi);
    // ... dozens more services
}

ServiceManager is the Binder registry — the phone book of every system service. Every service registers itself with a string name. Clients look up services by that name to get the Binder proxy.

getSystemService() — What Actually Happens

getSystemService(Context.AUDIO_SERVICE)
    → ContextImpl.getSystemService("audio")
    → SystemServiceRegistry.getSystemService()   ← static lookup map
    → factory creates AudioManager with IBinder
        → IBinder from ServiceManager.getService("audio")
                → this IBinder is a BINDER PROXY to AudioService in system_server
                → NOT a local AudioService object — a remote reference
    → every AudioManager method call crosses Binder into system_server
⚡ The Key Insight
You never hold the actual service object — only a proxy

When you call audioManager.setStreamVolume(), that call serializes into a Parcel, crosses a Binder boundary, arrives at AudioService in system_server on a Binder thread, executes, and the result is serialized back. Every. Single. Call. This is why calling system services on the main thread with slow operations causes ANR.

onBootPhase — Service Startup Ordering

Services can't all start doing work immediately — some depend on others. onBootPhase() lets services wait for the right moment:

PhaseWhenSafe to Do
PHASE_WAIT_FOR_DEFAULT_DISPLAYDisplay is readyAnything needing display — WindowManager, SurfaceFlinger
PHASE_LOCK_SETTINGS_READYSettings DB readableReading user preferences
PHASE_SYSTEM_SERVICES_READYCore services readyCalling other system services safely
PHASE_BOOT_COMPLETEDUser sees launcherEverything — this is the safe "fully booted" signal
📺 STB Context
Why your STB service must wait for PHASE_BOOT_COMPLETED

A custom STB service that tries to call HdmiControlService during onStart() will often get null or crash — HdmiControlService may not have registered yet. Always wait for PHASE_BOOT_COMPLETED before calling other services. This is the most common mistake when writing a new system service on STB.

5 Anchor Statements · System Services
01system_server is the single process that owns all shared Android resources. WMS, AMS, AudioService, HdmiService all live here.
02getSystemService() returns a Binder proxy, not the actual service object. Every call crosses a process boundary into system_server.
03ServiceManager is the Binder registry — every system service registers by name. Clients look up the Binder handle by that name.
04Services start in phases — never call other services in onStart(). Wait for PHASE_BOOT_COMPLETED to guarantee dependencies are ready.
05All services inside system_server share one Binder thread pool of 16 threads — a slow AudioService call consumes threads that WMS needs too.
system_server is the gatekeeper of all shared Android resources. getSystemService() gives you a Binder proxy — not the real object. Every method call crosses a process boundary. Services start in phases; call other services only after PHASE_BOOT_COMPLETED.
Chapter 03

HAL & Project Treble

From C-style dlopen() chaos to crash-isolated vendor processes

WHY HAL Exists — The Real Reason

Everyone says HAL exists to "isolate hardware differences." That's true but incomplete. The real reason is legal.

Android is open source (AOSP). But hardware drivers — the code that talks to Broadcom's Wi-Fi chip, Amlogic's video decoder, MediaTek's SoC — are proprietary. OEMs do not want to open-source their hardware code.

Without HAL, vendor code would be directly linked into the Android framework — GPL-tainted. OEMs couldn't keep anything private. HAL is the clean binary interface that lets Android framework stay open source while vendor implementations stay proprietary blobs.

🔌 The Power Socket Analogy
HAL is the standard socket. Vendor code is the plug.

The power socket (HAL interface) is a published standard. Anyone can manufacture a plug (vendor implementation) that fits it. The socket doesn't care what's inside the plug. The plug manufacturer doesn't care what's inside the wall. The interface is the contract. Both sides honor it without knowing each other's internals.

Old World — C-Style HAL (Pre Android 8)

Old way — HAL loaded directly into framework process:
framework process (system_server)
    → dlopen("audio.primary.amlogic.so")  ← vendor code loaded INTO system_server
    → calls function pointers directly
    → vendor code runs IN THE SAME PROCESS

// If vendor audio HAL has a bug and crashes:
// → system_server crashes
// → entire Android OS dies
// → device reboots
// → customer watching live TV sees black screen

This happened constantly on OEM devices before Android 8. Bad vendor HAL code destabilized the entire OS. On STBs this is unacceptable — a customer can't see "Android is restarting" during live TV.

Android 8 — Project Treble + HIDL

Google introduced Project Treble to fix this. The insight: run HAL implementations in their own isolated vendor processes.

// New world — HAL in separate vendor process:
system_server process          vendor HAL process
─────────────────────          ──────────────────────────
AudioService            ←HIDL→ audio.primary@2.0-service
    ↓                               ↓
AudioFlinger            ←HIDL→ AudioHAL implementation
                                    ↓
                              kernel ALSA driver

// If vendor audio HAL crashes now:
// → only the vendor process dies
// → system_server survives
// → Android can restart just the HAL process
// → 2-second audio hiccup, not a full reboot

Android 11+ — AIDL HAL (Current Standard)

HIDL worked but was a separate language and toolchain. Android 11 switched to AIDL HAL — using the exact same AIDL you already know from Binder, now also used for HAL interfaces.

EraMechanismProblem It Solved
Pre Android 8C-style dlopen()Nothing — HAL ran in same process, crashes killed system_server
Android 8–10HIDLCrash isolation via vendor processes. Separate toolchain.
Android 11+AIDL HALSame isolation, single toolchain. What you see in modern AOSP.

The Vendor Partition — Physical Separation on Your STB

Project Treble's partition separation is physical on your STB's eMMC storage:

/system   ← AOSP framework (platform team) — OTA updatable independently
/vendor   ← HAL implementations, kernel modules (chip vendor — Amlogic/Broadcom)
             rarely updated, proprietary binary blobs
/product  ← OEM apps, customizations — most frequently updated
📺 STB Context
Why STB OTA updates are more complex than phone updates

On a phone OTA, Google updates /system. The vendor (Qualcomm) already shipped a HAL that declares compatibility via VINTF manifest. The framework checks at boot — "does the vendor HAL support the interface version I need?" If yes, boot succeeds. On STB, the platform team must verify this compatibility matrix every time they update the framework. A version mismatch = boot failure. This is why STB OTA testing is longer and more careful than standard app updates.

5 Anchor Statements · HAL & Project Treble
01HAL is a legal firewall — framework stays open source, vendor implementation stays a proprietary binary blob. Both meet at the HAL interface.
02Pre-Treble: vendor HAL ran inside system_server. A HAL crash killed the entire OS. Post-Treble: vendor process is isolated — HAL crash = 2-second hiccup, not a reboot.
03HIDL (Android 8) → AIDL HAL (Android 11+). Same crash isolation goal, one unified toolchain instead of two.
04/system, /vendor, /product are physically separate partitions with different update cycles. Treble made them independently updatable.
05VINTF manifest declares which HAL versions the vendor supports. Framework checks compatibility at boot — version mismatch = boot failure.
HAL exists for legal isolation as much as technical abstraction. Project Treble moved vendor HAL code into isolated processes — a HAL crash no longer reboots the device. /system and /vendor are physically separate partitions, independently updatable.
Chapter 04

Flingers & HAL

Policy vs execution — who actually talks to hardware

The Pattern You Already Know

From Phase 2 you know: WMS is policy, SurfaceFlinger is execution. The same pattern repeats everywhere in Android. It's not coincidence — it's a deliberate architectural principle.

Service (Policy)Lives InFlinger (Execution)Lives In
WindowManagerServicesystem_serverSurfaceFlingersurfaceflinger process
AudioServicesystem_serverAudioFlingeraudioserver process
⚡ The Critical Rule
system_server services never talk to HAL. Flingers do.

AudioService and WindowManagerService handle policy only — permissions, routing decisions, who owns what. They never touch hardware. AudioFlinger and SurfaceFlinger own the data pipeline and talk directly to their respective HALs.

AudioService vs AudioFlinger — Full Separation

AudioService (system_server) — POLICY ONLY
  ✓ Manages volume levels
  ✓ Manages audio focus (who gets to play)
  ✓ Manages routing rules (speaker vs HDMI vs headphone)
  ✓ Enforces permissions
  ✗ Never touches a single audio sample

AudioFlinger (audioserver) — EXECUTION ONLY
  ✓ Actually mixes audio streams together
  ✓ Talks to Audio HAL
  ✓ Sends PCM data to hardware
  ✓ Manages audio threads and buffers
  ✗ Never makes policy decisions

Full Call Chain — Volume Change on STB

Your App
    audioManager.setStreamVolume(STREAM_MUSIC, 5, 0)
            │
            ▼  ← Binder → system_server
    AudioService.setStreamVolume()
            │  enforceCallingPermission(MODIFY_AUDIO_SETTINGS)
            │  updates internal volume table
            ▼  ← Binder → audioserver process
    AudioFlinger.setMasterVolume()
            │
            ▼  ← AIDL HAL → vendor process
    Audio HAL implementation
            │
            ▼  ← ioctl() → kernel
    kernel ALSA driver
            │
            ▼
    hardware DAC — volume actually changes

Full Call Chain — Audio Playback on STB

Your App
    AudioTrack.write(pcmData)
            │
            ▼  ← Binder → audioserver (bypasses system_server entirely)
    AudioFlinger.write()   ← playback goes DIRECT to AudioFlinger
            │
            ▼  ← AIDL HAL → vendor process
    Audio HAL
            │
            ▼
    kernel ALSA driver → hardware DAC → sound
💡 Key Observation

For actual audio playback, AudioTrack talks directly to AudioFlinger — system_server is not involved at all. system_server (AudioService) is only needed for policy decisions like volume changes, audio focus, and routing. The data pipeline bypasses it entirely for performance.

Why Flingers Are Separate Processes — Two Reasons

Stability: Audio mixing and surface compositing are real-time, complex operations. A bug in mixing code could crash the process. If AudioFlinger was inside system_server, an audio bug = full device reboot. In its own process, it crashes and restarts without touching system_server.

Performance: Flingers need real-time thread priority and tight VSYNC/audio clock coupling. Running inside system_server alongside hundreds of other services makes guaranteed scheduling impossible. Their own process = their own scheduling control.

5 Anchor Statements · Flingers & HAL
01system_server services own policy. Flingers own the data pipeline. WMS/AudioService never touch hardware. SurfaceFlinger/AudioFlinger talk directly to HAL.
02AudioFlinger lives in audioserver — a separate process from system_server with its own independent 16-thread Binder pool.
03AudioTrack playback goes directly to AudioFlinger, bypassing system_server entirely. AudioService is only involved for policy — volume, focus, routing.
04Flingers are separate processes for two reasons: stability (crash isolation from system_server) and performance (real-time scheduling control).
05HAL only ever talks to Flingers — never to system_server directly. Policy layer and hardware layer are deliberately kept apart.
AudioService and WMS own the rules. AudioFlinger and SurfaceFlinger own the data. HAL only ever sees the Flingers. This policy-vs-execution split is the same pattern everywhere in Android — intentional, repeated, and load-bearing.
Chapter 05

Binder Limits

The 1MB receiver buffer and the 16-thread pool — what they actually mean

Limit 1 — The 1MB Binder Buffer

The 1MB buffer is allocated on the receiving side. It belongs to the process that receives the Binder call — not the sender.

📬 The Mailbox Analogy
The mailbox belongs to the receiver, not the sender

You (sender) can write as many letters as you want. But the mailbox (1MB buffer) belongs to the recipient's house. If multiple senders are all stuffing letters into the same mailbox at the same time, the mailbox can overflow. The senders' capacity is irrelevant — it's the receiver's mailbox that's the bottleneck.

WRONG mental model (what most people think):
App 1 has its own 1MB budget for outgoing calls
App 2 has its own 1MB budget for outgoing calls
→ They never conflict

CORRECT mental model:
system_server has ONE 1MB receive buffer
ALL apps sending data TO system_server share this buffer

App 1 → sends 600KB → lands in system_server's 1MB buffer (600KB consumed)
App 2 → sends 500KB → lands in system_server's 1MB buffer (500KB more needed)
                       ─────────────────────────────────────────────────────
                       1100KB > 1MB  ❌  TransactionTooLargeException

Both Apps 1 and 2 are sending to the same destination — system_server. They fill the same 1MB inbox simultaneously. That's when the exception hits.

💡 The Return Direction Too

When your service returns data back to the caller — roles reverse. Now your app is the receiver. The returned data lands in your app's 1MB receive buffer. TransactionTooLargeException can throw in either direction. The rule is always the same: whoever is receiving at that moment owns the 1MB buffer.

ScenarioWhose buffer?Risk
App sends large Parcel to system_serversystem_server's 1MBShared with all other apps calling system_server simultaneously
system_server returns large data to your appYour app's 1MBOnly your app is receiving — lower risk but still a hard limit
App 1 sends to system_server, App 2 sends to AudioFlingerDifferent buffersNo conflict — different receiving processes, different mailboxes
⚠️ STB Advice
Never pass large data over Binder — especially during boot

During STB boot, many services start simultaneously and all register listeners, request window state, query packages. All those calls compete for system_server's 1MB buffer. Safe rule: keep any single transaction well under 100KB. For large data (Bitmaps, channel lists, EPG data), use ParcelFileDescriptor or shared memory (MemoryFile) — these pass a file descriptor over Binder, not the actual data.

Limit 2 — The 16 Binder Thread Pool

Every process that participates in Binder IPC gets its own pool of up to 16 threads to receive incoming Binder calls. Each process's pool is completely independent.

Each process has its own independent pool:
system_server process     → 16 Binder threads (own pool)
audioserver process       → 16 Binder threads (own pool)
Vendor HAL process        → 16 Binder threads (own pool)
Your App process          → 16 Binder threads (own pool)
SurfaceFlinger process    → 16 Binder threads (own pool)
Each is independent. One pool exhausted has zero effect on others.
⚡ Important Correction
Main thread is NOT one of the 16

The 16 Binder threads are a separate pool dedicated to handling incoming IPC. Your main thread is only involved when you make an outgoing synchronous Binder call from it — in which case it blocks waiting for the response. But it does not consume one of the 16 incoming slots.

What system_server's 16 Threads Actually Serve

system_server's 16 threads serve ALL incoming calls from ALL clients:

App 1  → calling AudioService        → picks Binder thread #1
App 2  → calling WindowManager       → picks Binder thread #2
App 3  → calling PackageManager      → picks Binder thread #3
App 4  → calling ActivityManager     → picks Binder thread #4
App 5  → calling AudioService again  → picks Binder thread #5
...
App 16 → calling anything            → picks Binder thread #16
                                        ──────────────────────
App 17 → calling anything            → ❌ all 16 busy — BLOCKS
                                        waits in kernel until one thread finishes

App 17 is not rejected. It blocks in the kernel until a thread frees. But if all 16 threads are blocked on slow operations — database writes, HAL calls, heavy computation — App 17 waits. App 17's main thread is frozen. If it waits more than 5 seconds — ANR.

💡 Threads Are Per-Call, Not Per-Process

App 1 can consume 3 threads simultaneously if it makes 3 parallel Binder calls. The pool doesn't reserve one thread per client process — it assigns threads per active incoming call, first come first served.

5 Anchor Statements · Binder Limits
01The 1MB buffer belongs to the receiver, not the sender. Whoever is receiving at that moment owns the limit — direction doesn't matter.
02Multiple apps sending to the same receiver simultaneously share one 1MB buffer. App 1 sending 600KB + App 2 sending 500KB to system_server = overflow.
0316 Binder threads is per process, not per-system. system_server has 16. audioserver has 16. Your app has 16. Each pool is completely independent.
04The main thread is NOT one of the 16. The 16 are a dedicated incoming IPC pool. Main thread is only involved when it blocks on an outgoing synchronous call.
05All services inside system_server — WMS, AMS, AudioService, HdmiService — share the same 16 threads. A slow AudioService call steals threads from WMS.
1MB is the receiver's inbox — shared by all simultaneous callers. 16 threads is a per-process pool for incoming calls — not per client, not per service within a process. Exhaust either limit and callers block silently until capacity frees.
Chapter 06

Production Patterns

RemoteCallbackList, oneway, HAL thread rules — the three things platform engineers get wrong

Pattern 1 — RemoteCallbackList

When a client registers a callback listener with your system service, what they pass over Binder is a proxy object — a "phone number" back into their process. Your service stores it and calls it when events happen.

Now the client app crashes. The process is gone. But your service still has the old proxy. When you call listener.onHdmiChanged(true) — you're calling a dead number. That's DeadObjectException.

// ❌ Plain list — dead clients cause exceptions, no cleanup
private val listeners = mutableListOf<IHdmiListener>()

fun notifyHdmiChanged(connected: Boolean) {
    listeners.forEach { listener ->
        listener.onHdmiChanged(connected)  ← client process died → DeadObjectException
        // exception stops the loop — remaining live clients never notified
    }
}
// ✅ RemoteCallbackList — production correct pattern
private val listeners = RemoteCallbackList<IHdmiListener>()

fun registerListener(listener: IHdmiListener) {
    listeners.register(listener)
    // RemoteCallbackList internally calls linkToDeath() on this Binder
    // When client process dies, kernel fires death notification
    // RemoteCallbackList removes the dead entry automatically
    // You never deal with DeadObjectException in broadcast
}

fun notifyHdmiChanged(connected: Boolean) {
    val count = listeners.beginBroadcast()
    try {
        repeat(count) { i ->
            listeners.getBroadcastItem(i).onHdmiChanged(connected)
        }
    } finally {
        listeners.finishBroadcast()  ← always call this, even if exception thrown
    }
}
🔍 How linkToDeath() Works Under the Hood

linkToDeath() is a method on IBinder that registers a DeathRecipient callback with the Binder kernel driver. When the process owning that Binder dies, the kernel fires the notification synchronously on a Binder thread. RemoteCallbackList uses this to remove dead entries automatically — before you ever try to broadcast to them. Every system service in AOSP that manages callbacks uses RemoteCallbackList for exactly this reason.

Pattern 2 — oneway

Every AIDL method is synchronous by default. The caller's thread blocks until your service finishes and returns. For fire-and-forget operations — logging, telemetry, state notifications — there is no reason to block the caller.

🍽️ The Restaurant Order Analogy
oneway = you place the order and go sit down

Without oneway: You stand at the counter frozen while the chef cooks. The counter is blocked. Nobody else can order.

With oneway: You place the order, get a buzzer, go sit down. The counter is free immediately for the next customer. You'll get the food when it's ready — but you didn't waste your time standing there.

In your AIDL interface:

// ❌ Caller blocks until logging is done — wastes their thread
void logChannelSwitch(int channelId);

// ✅ Caller returns immediately — fire and forget
oneway void logChannelSwitch(int channelId);
// ❌ Without oneway — on main thread
stbEventService.logChannelSwitch(channelId)
// if logging takes 200ms on the service side:
// main thread frozen 200ms → during channel switch → visible stutter
// block for 5 seconds → ANR
⚠️ The oneway Trade-off — Read This Before Using It
oneway has an unbounded queue. No backpressure.

oneway calls are queued on the service side. If you fire 1000 oneway calls rapidly and the service is slow processing them, they queue up. There is no backpressure — the queue can grow unbounded. Memory pressure, delayed processing, stale events.

Rule: oneway is the right tool for infrequent fire-and-forget events — channel switch telemetry, HDMI hotplug notifications, parental event logging. It is the wrong tool for high-frequency data streaming — audio samples, video frames, continuous sensor data.

Use Caseoneway?Why
Log channel switch event✅ YesInfrequent, fire-and-forget, no return value needed
Notify HDMI hotplug✅ YesRare event, caller doesn't need to wait for processing
Get current resolution❌ NoNeeds a return value — oneway can't return anything
Stream audio PCM data❌ NoHigh-frequency — queue grows unbounded under load
Update parental PIN❌ NoNeeds confirmation the write succeeded

Pattern 3 — HAL Thread Rule

HAL runs on a dedicated thread managed by the vendor HAL process. When the HAL calls back into your framework code on a hardware event — HDMI plugged, audio route changed — that callback arrives on the HAL's own thread.

The rule: Get off the HAL thread as fast as possible. Receive the event, post it somewhere, return immediately.

// ❌ Doing heavy work on HAL callback thread
override fun onHdmiEvent(event: HdmiEvent) {
    database.logHdmiEvent(event)           ← database write — 50ms, HAL thread blocked
    displayService.updateResolution(event) ← nested Binder call — HAL thread blocked
    currentHdmiState = event.connected     ← race condition — no lock
    // HAL thread is blocked for all of this
    // Next hardware event can't be processed until you return
    // Missed HDMI events, missed IR keys, stale state
}
// ✅ Return immediately — post to your own thread
override fun onHdmiEvent(event: HdmiEvent) {
    mainHandler.post {
        processHdmiEvent(event)  ← your actual logic runs off the HAL thread
    }
    // onHdmiEvent returns here — HAL thread is free immediately
}

private fun processHdmiEvent(event: HdmiEvent) {
    // Now on your own thread — do whatever you need safely
    database.logHdmiEvent(event)
    displayService.updateResolution(event.resolution)
    synchronized(stateLock) { currentHdmiState = event.connected }
    notifyListeners(event.connected)
}
💡 Same Rule as BroadcastReceiver.onReceive()

The HAL thread discipline is identical to what you already know about BroadcastReceiver.onReceive() — you're borrowing a thread you don't own. Do nothing blocking. Post work elsewhere. Return immediately. The consequence of blocking is also similar — queued events back up, and on HAL threads, that means dropped hardware events.

5 Anchor Statements · Production Patterns
01Always use RemoteCallbackList for managing callback listeners in system services. It calls linkToDeath() automatically and removes dead clients before you broadcast.
02linkToDeath() registers a kernel-level death notification — when the client process dies, the kernel fires it. Not polling, not timeout — immediate kernel-level signal.
03oneway returns immediately but has an unbounded queue with no backpressure. Right for infrequent fire-and-forget. Wrong for high-frequency data streaming.
04Get off the HAL thread immediately. Receive the event, post to your own Handler, return. Blocking the HAL thread delays all subsequent hardware events.
05HAL thread discipline = BroadcastReceiver.onReceive() discipline — you are borrowing a thread you don't own. Never block it.
Three production patterns every platform engineer must know: RemoteCallbackList for safe multi-client callbacks with automatic dead-client cleanup, oneway for fire-and-forget with awareness of its unbounded queue, and HAL thread hand-off to avoid starving the hardware event pipeline.
