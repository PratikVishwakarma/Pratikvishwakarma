Chapter 01

WMS Architecture

The single authority over everything on screen

WHY WindowManager Exists

Your STB has one screen. Multiple apps and services all want to draw on it simultaneously. Someone needs to be in charge — deciding who draws where, who sits on top of whom, and who receives the remote's key presses.

WindowManager is that someone. It is a system service running in system_server that acts as the single authority over everything visible on screen. No app draws directly to the display. Every app negotiates with WindowManager first.

🎬 The Cinema Multiplex Analogy
WMS is the multiplex manager
  • Screen = the physical STB display
  • Each movie = a window (your app, your overlay, system UI)
  • Manager (WMS) = decides which movie plays where, at what layer
  • Ticket checker = input dispatch — only the focused window gets the remote

The movies don't talk to each other. They only talk to the manager. On STB: Launcher, Settings overlay, system status bar — all coordinated by one WMS.

Three Components — Each Has One Job

YOUR APP          →        WMS (system_server)        →      SurfaceFlinger
─────────────────────        ──────────────────────────        ──────────────────
"I want to show       →    "Approved. Layer 5,          →   "Got all layers.
 a window here"             size 400x300, you               Compositing into
                               get input"                       one final frame"

  POLICY decisions               EXECUTION only
ComponentLives InJob
ViewRootImplYour app processBridge between your View tree and WMS. Every window has exactly one. Owns the Surface your app draws into.
WindowManagerServicesystem_serverMaintains WindowState per window. Decides z-order, size, permissions, who gets input.
SurfaceFlingerSeparate processTakes all Surface buffers, composites them into the final frame sent to display hardware. Never makes policy decisions.
⚡ The Key Insight
WMS is policy. SurfaceFlinger is execution. They are deliberately separate.

WMS never touches pixel data. SurfaceFlinger never makes decisions. This lets you reason about each independently — and it's why overlay z-ordering and rendering performance are separate problems.

5 Anchor Statements · WMS Architecture
01WMS is the single authority over the screen — no app draws or receives input without WMS approving it first.
02WMS decides policy (who, where, z-order, input). SurfaceFlinger composites pixels. Deliberately separate.
03Every window lives in two places — a local object in your app, and a WindowState inside WMS. Mutating the local object does nothing.
04Window type determines z-order. TYPE_APPLICATION_OVERLAY sits above normal apps — WMS enforces this by rule, not by your code being careful.
05Input goes to the topmost focusable window. A visual-only overlay without FLAG_NOT_FOCUSABLE silently breaks D-pad.
WindowManager is the kernel of the visual system — the single gatekeeper between processes and the display. WMS owns the rules; SurfaceFlinger executes them. Every window on your STB screen is there because WMS approved it.
Chapter 02

WindowManager Policy

The rulebook WMS follows — and OEMs override

WHY Policy is Separate from WMS

WMS knows how to manage windows. But it needs someone to tell it what the rules are. Should pressing Power lock the screen or shut down? Which windows show above the keyguard? What orientation should we force on STB?

These rules change per device. A phone behaves differently from a STB. WindowManagerPolicy is that rulebook — an interface that OEMs like Reliance Jio override to customize behavior without forking WMS.

⚖️ The Judge & Rulebook Analogy
WMS is the judge. WindowManagerPolicy is the rulebook.

The judge (WMS) runs the proceedings — manages windows, routes input, handles layout. But the judge follows a rulebook. On stock Android, Google wrote it. On STB, product company hands the judge a custom rulebook — same courtroom, different rules. Same WMS mechanics, OEM-specific behavior.

PhoneWindowManager — The Default Implementation

Despite the name, PhoneWindowManager runs on all Android devices including STBs. Three main responsibilities:

ResponsibilityWhat It DoesSTB Example
Key InterceptioninterceptKeyBeforeDispatching() — system gets first look at every key event before any appGuide, Record, EPG, color buttons consumed here. Apps never see them.
Rotation PolicyDecides if rotation is allowed and which orientation to useSTB locked to landscape — portrait requests from apps are ignored at policy level
Keyguard PolicyWhich windows show above lock screen, when to dismiss keyguard, idle timeout behaviorParental lock screen, screensaver after idle, PIN overlay
Key interception flow:
Remote button press
    → InputDispatcher
    → PhoneWindowManager.interceptKeyBeforeDispatching()
          ← system decides: consume this or pass to app?
    → if consumed: action taken, app never sees event
    → if not consumed: event delivered to focused window
📺 STB Context
Why some remote buttons "just work" everywhere

Press Guide from Netflix, STB Settings, or any app — EPG opens. That global behavior isn't magic. Jio's AOSP team overrides interceptKeyBeforeDispatching() and handles the Guide keycode before any app process knows a button was pressed. This is also why apps cannot remap those buttons — the system consumes them first.

5 Anchor Statements · WindowManagerPolicy
01WMS handles mechanics. WindowManagerPolicy handles rules. Separated so OEMs can customize without touching WMS itself.
02PhoneWindowManager is the default rulebook — runs on phones, tablets, STBs. OEMs override it for device-specific behavior.
03Key interception happens at the policy layer before any app sees the event. Home, Power, STB special buttons consumed here.
04Rotation, keyguard behavior, idle/screensaver logic — all policy decisions in PhoneWindowManager.
05Any STB remote button that works system-wide regardless of which app is open — WindowManagerPolicy is intercepting it first.
WindowManagerPolicy is the customization seam in WMS — where device makers inject their rules. PhoneWindowManager is the default; OEMs extend it. Every global remote button on STB goes through here before any app is involved.

Z-Order Layers

Window types, visual hierarchy, and where your overlay sits

WHY Window Types Matter

WMS assigns every window a layer based on its type. This is a fixed menu — not negotiated at runtime. The type you declare in LayoutParams determines your position in the stack. Always.

Real Examples — Each Layer

LayerReal STB / Android Example
Secure System OverlayParental lock screen enforced by platform. Your overlay service cannot draw above this — WMS rejects it by window type.
System UIStatus bar (time, signal, volume). Volume HUD when you press remote volume button. These are AOSP's SystemUI app — not your code.
Your Overlay Service"Recording..." badge, CC panel sliding in, PIP frame, parental PIN dialog from your background service.
Normal App WindowAny Activity — STB Settings, the OTT player, the launcher home screen.
WallpaperThe animated or static background behind the launcher. Everything else renders on top of it.

Toasts — The Special Case

Toasts sit between overlay service and System UI. Before Android 11 they used TYPE_TOAST directly — apps abused this for fake system-looking phishing messages. Now they route through NotificationManagerService with rate limiting.

⚠️ STB Advice
Don't use Toast for production STB UI from a background service

Toast is fire-and-forget — no control over timing, position, or style. Use your own TYPE_APPLICATION_OVERLAY window instead. You own the lifecycle, the animation, and the dismiss logic.

💡 Key Rule

Any STB overlay based work lives in the third row — always. WMS guarantees it floats above apps but can never accidentally cover System UI or the lock screen. That boundary is enforced by window type, not by your code being careful.

Chapter 04

SurfaceControl & Transactions

Atomic layer updates, VSYNC sync, no-jank animation

WHY updateViewLayout() Causes Jank

updateViewLayout() sends updated window params to WMS over Binder — one call per change. Fine for occasional repositioning. But animating at 60fps means 60 Binder calls per second — each with serialization overhead, thread scheduling, and kernel crossings. Frames miss their VSYNC deadline. Jank appears.

🗄️ The Database Transaction Analogy
SurfaceControl.Transaction = a database transaction for the screen

Without transaction: UPDATE position on frame 1, UPDATE opacity on frame 2, UPDATE size on frame 3 — user sees intermediate states, visual glitch.

With transaction: BEGIN → set position + set opacity + set size → COMMIT. All three applied on the same VSYNC frame. Clean, atomic, no flash.

What SurfaceControl Is

Every window WMS creates has a corresponding layer in SurfaceFlinger. SurfaceControl is the handle to that layer. It represents position, size, alpha, visibility, z-order, and transform. WMS owns it for normal windows — but as a system app on STB, you can use it directly.

APIWhat It DoesWhen to Use
updateViewLayout()Binder call to WMS, one change at a timeStatic position/size changes. Not animation.
Transaction.commit()Async, synced to VSYNC boundaryAnimation — smooth, no Binder per frame
Transaction.apply()Synchronous, applied immediatelyInstant one-shot changes
ViewPropertyAnimatorUses SurfaceControl transactions internallyEasiest path — framework handles VSYNC sync

Practical Usage on STB

Option 1 — ViewPropertyAnimator (Simplest)

// CC panel slide-in — framework uses SurfaceControl.Transaction internally
overlayView.translationX = panelWidth.toFloat()
overlayView.animate()
    .translationX(0f)
    .setDuration(300)
    .setInterpolator(DecelerateInterpolator())
    .start()  // smooth, VSYNC-synced, no Binder per frame

Option 2 — Direct Transaction (System App)

// Alpha fade — each commit() synced to VSYNC, zero Binder overhead
ValueAnimator.ofFloat(0f, 1f).apply {
    duration = 300
    addUpdateListener { animator ->
        val alpha = animator.animatedValue as Float
        SurfaceControl.Transaction()
            .setAlpha(surfaceControl, alpha)
            .commit()  // async, VSYNC boundary
    }
    start()
}
⚠️ The Jank Pattern to Avoid
// ❌ 60 Binder calls/second — jank guaranteed at scale
ValueAnimator.ofInt(startX, 0).apply {
    addUpdateListener {
        params.x = it.animatedValue as Int
        windowManager.updateViewLayout(overlayView, params)  ← Binder every frame
    }
    start()
}
5 Anchor Statements · SurfaceControl
01updateViewLayout() crosses Binder every call — fine for static changes, causes jank for animation at 60fps.
02SurfaceControl is a handle to a layer in SurfaceFlinger. Every window has one — WMS owns it, SurfaceFlinger renders it.
03Transaction batches changes and commits them atomically at one VSYNC frame — like a database transaction for the screen.
04.commit() syncs to VSYNC automatically — use for animation. .apply() is immediate — use for instant changes.
05ViewPropertyAnimator uses SurfaceControl transactions internally — smooth animation without touching SurfaceControl directly.
updateViewLayout() is a Binder call — great for one-off changes, wrong for animation. SurfaceControl.Transaction commits all changes atomically at a VSYNC boundary, bypassing Binder overhead entirely. ViewPropertyAnimator is the practical entry point.
Chapter 05

Input Dispatch

From remote button press to your View — the complete path

The Simple Version — Start Here

When you press a button on the STB remote, Android has to figure out who gets that press — and it always follows the same path to find the answer.

📞 The Company Phone Call Analogy
Input dispatch is a company switchboard
  • Call arrives = button press
  • Receptionist screens it = PhoneWindowManager — "Is this for the CEO directly?" → Home/Guide/Power = yes, handled by system
  • Transferred to right department = focused window (WMS decides)
  • Rings the person at their desk = focused View gets the event
  • Nobody picks up = event dropped silently

Same rules. Every single time. No shortcuts.

The Three Things That Actually Matter for STB

RuleWhat It Means
System gets first pickHome, Guide, Record, Power — gone before your app wakes up. You cannot intercept these unless you're a system app overriding PhoneWindowManager.
WMS decides which windowTopmost focusable window wins. This is why your CC overlay was stealing D-pad — it was on top and focusable.
Inside your app, focus winsNo focused View = event dropped silently. The most common D-pad bug on STB. Always ensure something has focus.

Why D-pad Gets Lost — Three Real Scenarios

🐛 Scenario 1 — Overlay steals focus

Overlay added without FLAG_NOT_FOCUSABLE → WMS gives it focus → D-pad events go into overlay's empty View tree → nobody consumes → dropped silently. The app behind appears frozen.

🐛 Scenario 2 — No View focused inside the window

Window correctly focused by WMS. But no View inside has focus. D-pad event arrives, travels down the tree, nobody claims it. Fix: call focusRequester.requestFocus() when your screen appears.

🐛 Scenario 3 — Focus on a non-visible View

A View has focus but is scrolled off-screen or GONE. FocusFinder only considers visible on-screen Views. Focus appears stuck with no obvious reason or error.

🔗 ANR Connection
InputDispatcher's 5-second timer

InputDispatcher starts a timer when it sends an event. If the app doesn't acknowledge within 5 seconds, InputDispatcher triggers ANR. This is why a frozen main thread causes "Application Not Responding" — the event sits unacknowledged. Also: InputChannel uses a socket, not Binder — deliberately chosen for lower latency on high-frequency input events.

5 Anchor Statements · Input Dispatch
01Every key event travels the same path — kernel → InputReader → InputDispatcher → PhoneWindowManager → InputChannel → ViewRootImpl → focused View. No shortcuts.
02InputDispatcher asks WMS for the focused window on every event. WMS owns focus state — your app influences it through window flags and requestFocus().
03PhoneWindowManager intercepts before your app sees anything. STB remote buttons (Guide, Record, Home) consumed here — apps never get them.
04InputChannel is a socket, not Binder — deliberately chosen for lower latency on high-frequency input events.
05ANR is InputDispatcher's 5-second timer — frozen main thread = unacknowledged events = ANR. Unrelated to crashes, it's a dispatch timeout.
Input dispatch always follows one path — kernel to ViewRootImpl — with the system getting first pick at the policy layer. WMS owns focus state. No focused View means silent event drops. These three rules explain 90% of D-pad bugs on STB.
Chapter 06

Real STB Bugs

The CC overlay focus loss — full root cause analysis

The Bug — What Was Happening

Clicking a blank non-focusable area via mouse/ADB caused D-pad to stop working. Pressing Home brought the Launcher behind, but the CC overlay stayed visible. The STB appeared frozen.

The full bug chain:
Mouse click on blank non-focusable area
    → WMS routes event to CC overlay window
    → no View consumes it
    → focus goes to limbo
    → D-pad has no valid target → STB feels stuck
    → user presses Home
    → PhoneWindowManager intercepts → Launcher appears behind  ✓
    → CC overlay service never got a signal to hide
    → overlay stays visible over Launcher
    → looks completely broken

Two Separate Problems

Problem 1 — Focus stolen by blank area
Missing FLAG_NOT_FOCUSABLE on an interactive panel

The CC overlay was an interactive panel — it needed to be focusable. But clicking the dim area (outside panel bounds, inside window bounds) sent clicks to the overlay window. No View consumed them. Focus went to limbo. D-pad had no valid target.

Fix: Handle the dim area click explicitly — root view click listener calls hideOverlay(). Panel content view consumes its own clicks so they don't bubble up.

Problem 2 — Overlay didn't hide on Home
Service has no lifecycle — it doesn't know Home was pressed

TYPE_APPLICATION_OVERLAY windows are added by a background Service. The service has no onPause(), no onStop(). When the user presses Home, the system doesn't notify the service. The overlay stays until the service explicitly calls removeView().

Fix: Listen to ACTION_CLOSE_SYSTEM_DIALOGS in a BroadcastReceiver and call hideOverlay().

The Three Overlay Mistakes

Mistake 1 — Wrong window type

// ❌ TYPE_PHONE deprecated in API 26+
params.type = WindowManager.LayoutParams.TYPE_PHONE

// ✅ Correct for API 26+
params.type = WindowManager.LayoutParams.TYPE_APPLICATION_OVERLAY

Mistake 2 — Missing FLAG_NOT_FOCUSABLE on visual-only overlays

// ❌ Overlay steals D-pad — app behind silently frozen
params.flags = WindowManager.LayoutParams.FLAG_NOT_TOUCH_MODAL

// ✅ Visual-only pass-through overlay
params.flags = (WindowManager.LayoutParams.FLAG_NOT_FOCUSABLE
        or WindowManager.LayoutParams.FLAG_NOT_TOUCH_MODAL
        or WindowManager.LayoutParams.FLAG_NOT_TOUCHABLE)

Mistake 3 — Mutating LayoutParams without updateViewLayout()

// ❌ Does nothing — params mutation is local to your process
// WindowState in WMS is completely unaware
params.x = 100
params.y = 200

// ✅ Correct — pushes updated params to WMS via Binder
params.x = 100
params.y = 200
windowManager.updateViewLayout(overlayView, params)
🔍 Why updateViewLayout() Must Be Called

When you called addView(), WMS created a WindowState and copied your params into it in a different process. What you hold locally is a separate copy. Mutating it never crosses the process boundary. updateViewLayout() calls IWindowSession.relayout() over Binder — that's what actually updates the WindowState in WMS and repositions the SurfaceFlinger layer.

Two root causes: overlay stole focus because the dim area had no click handler, and overlay stayed because services have no lifecycle awareness of Home. Both are patterns you'll encounter again — fix flags for visual overlays, add receivers for service lifecycle awareness.

Production Code Patterns

releaseOverlay, hideOverlay, combined system receiver

The Pattern — Two Methods, One Responsibility Each

MethodCalled ByDoesAnalogy
releaseOverlay()hideOverlay() + onDestroy()All cleanup. Safe to call multiple times.Packing your bags
hideOverlay()User-triggered eventsCleanup + stops the serviceWalking out the door
onDestroy()System kills serviceSafety net — calls releaseOverlay()Fire alarm backup

releaseOverlay() — Single Source of Truth

/**
 * Single source of truth for all overlay cleanup.
 *
 * Defocuses window BEFORE removing — WMS correctly transfers
 * the focus token back to the underlying window.
 * Without this, WMS logs "Unknown focus tokens, dropping
 * reportFocusChanged" and underlying app loses D-pad input.
 *
 * Safe to call multiple times — null guards prevent double execution.
 */
private fun releaseOverlay() {
    try {
        composeView?.let { view ->
            // Defocus before remove — returns focus token to underlying window
            try {
                val defocusParams = view.layoutParams as? WindowManager.LayoutParams
                if (defocusParams != null) {
                    defocusParams.flags = defocusParams.flags or
                            WindowManager.LayoutParams.FLAG_NOT_FOCUSABLE
                    wm?.updateViewLayout(view, defocusParams)
                }
            } catch (_: Throwable) {}
            wm?.removeView(view)
        }
    } catch (_: Throwable) {}

    if (systemEventReceiverRegistered) {
        try { unregisterReceiver(systemEventReceiver) } catch (_: Throwable) {}
        systemEventReceiverRegistered = false
    }
    AppState.setCCVisible(false)
    bridgeService?.notifyCCStateChanged(false)
    owner?.let { it.onPause(); it.onStop(); it.onDestroy() }
    composeView = null; wm = null
}

hideOverlay() — User-Triggered Close

/**
 * Intentional user-triggered close.
 * Called on: Home press, screen off, screensaver, dim area tap, back.
 *
 * Flow: releaseOverlay → unbindService → stopSelf
 * onDestroy will call releaseOverlay again — null guards make it a no-op.
 */
fun hideOverlay() {
    releaseOverlay()
    try { unbindService(bridgeConnection) } catch (_: Exception) {}
    stopSelf()  // triggers onDestroy as safety net
}

override fun onDestroy() {
    super.onDestroy()
    releaseOverlay()  // safety net — null guards prevent double execution
    try { unbindService(bridgeConnection) } catch (_: Exception) {}
}

Combined System Event Receiver

// One receiver handles all "system told us to close" signals
private var systemEventReceiverRegistered = false
private val systemEventReceiver = object : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        when (intent.action) {
            Intent.ACTION_DREAMING_STARTED,
            Intent.ACTION_SCREEN_OFF -> {
                logger("OverlayService", "screen off/screensaver — closing CC")
                hideOverlay()
            }
            @Suppress("DEPRECATION")
            Intent.ACTION_CLOSE_SYSTEM_DIALOGS -> {
                // Deprecated API 31+ but works on system apps / custom AOSP
                logger("OverlayService", "home/recent pressed — closing CC")
                hideOverlay()
            }
        }
    }
}

// Register in onCreate()
val filter = IntentFilter().apply {
    addAction(Intent.ACTION_DREAMING_STARTED)
    addAction(Intent.ACTION_SCREEN_OFF)
    @Suppress("DEPRECATION")
    addAction(Intent.ACTION_CLOSE_SYSTEM_DIALOGS)
}
ContextCompat.registerReceiver(this, systemEventReceiver, filter,
    ContextCompat.RECEIVER_EXPORTED)
systemEventReceiverRegistered = true
💡 Why @Suppress at the call site

@Suppress("DEPRECATION") at the exact usage line is better than suppressing at class or file level. It makes clear exactly which line is intentionally using a deprecated API — and why — rather than silently hiding all deprecation warnings across the file.

The pattern is: one cleanup method (releaseOverlay) called from two places (user intent + system safety net). Never duplicate cleanup logic. Always defocus before removeView so WMS can return the focus token cleanly to the window behind.
Interview Q&A

Questions & Answers

Tap each question to reveal the answer
Walk me through what happens when a system overlay service adds a window on Android.+
The app calls windowManager.addView() with LayoutParams specifying TYPE_APPLICATION_OVERLAY. This crosses a Binder boundary into WindowManagerService in system_server. WMS checks SYSTEM_ALERT_WINDOW permission first. If approved, it creates a WindowState — its internal record — and assigns a layer based on window type, placing it above normal apps. It creates a SurfaceControl handle to a layer in SurfaceFlinger. The app now has a Surface to draw into. When the app draws, SurfaceFlinger composites that layer above the app behind it and the overlay appears on screen.
How does Android ensure the Home button always works even if an app is frozen?+
Key events pass through WindowManagerPolicy before reaching the focused app. Specifically interceptKeyBeforeDispatching() in PhoneWindowManager is called first on every key event. For the Home key, the policy consumes the event and triggers the launcher directly — the app never sees it. This is why no matter what state an app is in, Home always works. On STB, remote buttons like Guide and Record use the same mechanism — intercepted at the policy layer system-wide.
What is the relationship between WMS and SurfaceFlinger?+
WMS is the policy layer — it decides which windows exist, their z-order, size, and who gets input. SurfaceFlinger is the compositor — it takes the Surface buffers (one per window) and composites them into the final frame using GPU/hardware composer. They communicate via SurfaceControl transactions. WMS never touches pixel data; SurfaceFlinger never makes policy decisions. This separation is intentional — it lets each evolve independently.
Why does animating a window with updateViewLayout() cause jank, and how would you fix it?+
updateViewLayout() is a Binder call that crosses into WMS in system_server on every invocation. At 60fps that's 60 Binder round trips per second — each with serialization overhead, thread scheduling, and kernel crossings. Frames start missing their VSYNC deadline and jank appears. The fix is SurfaceControl.Transaction — it batches layer changes and commits them directly to SurfaceFlinger at a VSYNC boundary, bypassing Binder overhead entirely. For most overlay animations on STB, ViewPropertyAnimator is the practical choice since Android uses SurfaceControl transactions internally.
Walk me through what happens when a user presses D-pad DOWN on an Android TV remote.+
The remote sends a hardware scancode which the kernel exposes as an evdev event. InputReader picks it up, maps it to KEYCODE_DPAD_DOWN using the device's key layout file, and passes it to InputDispatcher. InputDispatcher first checks with PhoneWindowManager — if the system wants to consume it, it stops there. Otherwise InputDispatcher asks WMS for the currently focused window, serializes the KeyEvent, and sends it through that window's InputChannel — a socket connection to the app process. ViewRootImpl receives it and dispatches it down the View tree to the focused View. If no View consumes it, FocusFinder tries to move focus downward. If nothing handles it, the event is silently dropped.
Why would an overlay on a STB steal D-pad events unexpectedly?+
Missing FLAG_NOT_FOCUSABLE on the overlay window. InputDispatcher asks WMS for the focused window — WMS picks the topmost focusable window. If the overlay is on top and focusable, it becomes the focused window and receives all D-pad key events. If the overlay has no focusable Views to consume those events, they are dropped silently. The app behind appears frozen with no error, no crash, no log. Always add FLAG_NOT_FOCUSABLE to visual-only overlays.
What is a WindowState?+
WindowState is WMS's internal representation of a single window. It lives in system_server and tracks: the window's SurfaceControl (SurfaceFlinger layer handle), its LayoutParams (position, size, flags, type), its current frame/bounds, animation state, its parent ActivityRecord, and its InputChannel for receiving input events. Every visible window on screen — app, overlay, system UI — has exactly one WindowState in WMS.
