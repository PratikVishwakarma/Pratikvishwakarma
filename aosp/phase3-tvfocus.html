Chapter 01

WHY Focus Management Exists

The problem that touch solves automatically — but D-pad doesn't

The Core Problem

On a phone, you tap a button. The system knows exactly what you tapped — it has X and Y coordinates. No ambiguity. No decisions to make.

On a STB remote, there is no touch. The user presses RIGHT. The system now has one job: which view should highlight next?

That's the entire problem focus management is solving. It is a spatial navigation problem — given the current position and a direction, find the most appropriate next target.

🎮 The Grid Analogy
Focus is a cursor on an invisible grid

Think of the screen as a grid. On touch, your finger IS the cursor — it places itself exactly. On D-pad, there is one cursor and it moves in four directions. The system must figure out which cell to land on — and that decision isn't always obvious.

On STB, this invisible cursor is traversing your settings screen, your EPG grid, your OTT channel row — every single remote key press moves it one step.

Why It Gets Complex — A Real Layout

Consider this layout — focus is on Button A, user presses RIGHT:

  [Button A]          [Button B]
        [Button C]

Which view wins? B is directly right. C is below-right — closer horizontally but offset vertically. The answer depends on geometry + a scoring algorithm. Android's answer is FocusFinder — the class that makes this decision every time a D-pad key fires.

01Touch gives XY coordinates — D-pad gives only a direction. The system must resolve which view that direction points to. That's the focus problem.
02FocusFinder is the spatial brain — it runs every time a D-pad key fires and no view consumed it. It decides who gets focus next.

TV focus management exists because D-pad navigation is a spatial problem that touch solves automatically but remote controls don't. FocusFinder is Android's answer — a geometry-based scoring engine that decides which view deserves focus on every key press.
Chapter 02

FocusFinder Internals

The geometry scoring algorithm — and why it sometimes picks the wrong view

What FocusFinder Is

FocusFinder is a singleton (one per process) that lives at frameworks/base/core/java/android/view/FocusFinder.java. Its one method that matters is findNextFocus(root, focused, direction).

It runs only when no view consumed a key event. If your RecyclerView item handles DPAD_RIGHT internally — FocusFinder never runs. It only kicks in for the unhandled case: find the next logical view in this direction.

The Algorithm — Step by Step

Step 1 — Check explicit override first

// If you set this in XML or code, FocusFinder skips geometry entirely
android:nextFocusRight="@id/buttonB"

// Or programmatically:
view.nextFocusRightId = R.id.buttonB

Explicit wiring always wins. Geometry scoring only runs when no override exists.

Step 2 — Collect all focusable views

FocusFinder scans the entire View hierarchy and collects every view where isFocusable() == true and isShown() == true. Views that are GONE, INVISIBLE, or off-screen bounds are filtered out.

Step 3 — Filter by direction

Any view whose center falls in the opposite direction from the current focused view is discarded immediately. If you pressed RIGHT, anything to the left of the focused view's center is gone.

Step 4 — Score each candidate

For every remaining candidate, FocusFinder calculates a weighted distance score:

// Simplified from FocusFinder.java AOSP source
int getWeightedDistanceFor(int majorAxisDistance, int minorAxisDistance) {
    return 13 * majorAxisDistance * majorAxisDistance
             + minorAxisDistance * minorAxisDistance;
}
TermMeaningExample (pressing RIGHT)
majorAxisDistanceDistance in the direction you pressedHorizontal distance between view centers
minorAxisDistanceDistance perpendicular to directionVertical offset between view centers
13× multiplierStrongly penalises far-away views in directionA view directly right wins over one diagonally placed

Lowest score wins.

Scoring — Concrete Example

Focus is on Button A. User presses RIGHT. Two candidates: B (directly right) and C (below-right).

Directly to the right

617,732
Zero minor axis penalty — clean directional alignment
⚠️ Loser — Button C

Minor axis penalty from vertical offset hurts heavily
🔬 Why 13× on the Major Axis?
Empirical AOSP constant — prevents diagonal teleports

The 13× multiplier means: views far in your intended direction are penalised heavily. A view slightly off-axis (to the side) gets a large minor penalty. This combination ensures focus moves where you expect — straight in the direction pressed — and doesn't jump diagonally to a closer-but-wrong target.

The constant is not derived from first principles — it's a tuned empirical value that felt right across Android's test layouts. Verify it yourself at cs.android.com → search getWeightedDistanceFor in FocusFinder.java.

When FocusFinder Gets It Wrong

FocusFinder fails when geometry doesn't match intent. Classic STB example: an EPG grid where the next cell is partially off-screen (clipped, not laid out yet). FocusFinder skips it — the cell isn't visible. The next scored candidate is three rows away but has a lower geometric score. Focus teleports.

⚠️ Fix for Wrong Scoring
Use explicit wiring — bypass geometry entirely
// XML
android:nextFocusRight="@id/correctCell"

// Programmatic
view.nextFocusRightId = R.id.correctCell

// Override in ViewGroup — most powerful
override fun focusSearch(focused: View, direction: Int): View? {
    if (direction == FOCUS_RIGHT && focused == channelNameView) {
        return epgGridFirstCell
    }
    return super.focusSearch(focused, direction)
}
5 Anchor Statements · FocusFinder
01FocusFinder runs only for unhandled key events. If a view consumes the key, FocusFinder never runs.
02Explicit nextFocusXxx always beats geometry. Set it in XML or code to bypass scoring entirely — the most reliable fix for wrong-target bugs.
03Score = 13 × major² + minor². Lowest score wins. The 13× penalises views far in the pressed direction; minor axis handles ties.
04FocusFinder only considers visible, laid-out views. Off-screen or GONE views are invisible to it — this is why EPG cells get skipped.
05Override focusSearch() in a ViewGroup for full directional control — more powerful than per-view nextFocusXxx attributes.
FocusFinder uses weighted geometric distance to pick the next focused view — strongly favouring views aligned in the direction pressed. It works well for predictable layouts, fails on dynamic or partially-visible content. Use explicit nextFocusXxx or override focusSearch() as your escape hatch.
Chapter 03

View Focus Internals

requestFocus(), touch mode behaviour, and onFocusChanged

requestFocus() — The Full Chain

When FocusFinder picks a view, or you call requestFocus() manually, this chain runs internally:

view.requestFocus()
    └─▶ requestFocus(direction, previouslyFocusedRect)
         └─▶ requestFocusNoSearch()
              ├─▶ canTakeFocus()   ← visibility + focusable + enabled check
              ├─▶ isInTouchMode() && !isFocusableInTouchMode()  ← SKIPPED on STB
              └─▶ handleFocusGainInternal()  ← actually transfers focus
                   └─▶ onFocusChanged(gainFocus=true, ...)

Touch Mode — The STB Difference

On a phone, when you tap the screen the device enters touch mode. In touch mode, only views with focusableInTouchMode=true can receive focus programmatically. This prevents buttons from appearing focused after a touch — it would look weird.

On STB, isInTouchMode() always returns false — there is no touchscreen. This means the focusableInTouchMode restriction is skipped entirely. Any view with focusable=true can receive focus, regardless of its focusableInTouchMode setting.

📺 STB Context
Shared components behave differently on phone vs TV

If you write a shared component used on both mobile and STB, a view that is focusable=true but focusableInTouchMode=false will accept D-pad focus on STB but refuse programmatic focus on phone. This is intentional per-platform behaviour — not a bug. Verify with view.isInTouchMode() if you need to handle both.

onFocusChanged() — Your Visual Reaction Hook

This callback fires every time focus arrives or leaves a view. It is where you drive your focus highlight — scale animation, border change, color shift. On TV, if the user cannot see which item is focused, the entire screen feels broken regardless of how correct the logic is.

// Classic View
override fun onFocusChanged(gainFocus: Boolean, direction: Int, previouslyFocusedRect: Rect?) {
    super.onFocusChanged(gainFocus, direction, previouslyFocusedRect)
    animate()
        .scaleX(if (gainFocus) 1.06f else 1.0f)
        .scaleY(if (gainFocus) 1.06f else 1.0f)
        .setDuration(150)
        .start()
}

ViewGroup — onRequestFocusInDescendants()

When a ViewGroup receives a focus request (not a specific child — the group itself), it delegates to:

// ViewGroup.java
protected boolean onRequestFocusInDescendants(int direction, Rect previouslyFocusedRect)

Default behaviour: focus the first focusable child in the given direction. You can override this to restore a "last focused position" — which is exactly what RecyclerView does internally, and why it can trap focus (covered in Chapter 04).

5 Anchor Statements · View Focus Internals
01requestFocus() fails silently if the view is not visible, not focusable, or not enabled. No error, no exception — just returns false.
02On STB, isInTouchMode() is always false. focusableInTouchMode is irrelevant on STB — any focusable view can receive focus.
03onFocusChanged() is where your UI reacts — scale, border, color. Missing this is why some STB screens look broken even when focus logic is correct.
04Always call requestFocus() inside .post {} — the view must be fully laid out first. Calling it directly in onCreate/onResume before layout often silently fails.
05ViewGroups delegate focus to children via onRequestFocusInDescendants(). Overriding this method is how you implement focus memory in custom containers.
requestFocus() is silent on failure — verify the view is visible, focusable, and enabled first. On STB, touch mode restrictions don't apply. onFocusChanged is your visual hook — missing it makes focus invisible to the user even when the system logic is perfect.
Chapter 04

The RecyclerView Focus Trap

Why D-pad gets stuck in a list — and three ways out

WHY RecyclerView Traps Focus

RecyclerView overrides onRequestFocusInDescendants(). When focus tries to escape downward, RecyclerView intercepts that request and says: "No — refocus my last focused child instead." The view below the RecyclerView never gets a chance.

This is intentional behaviour — it was designed to restore the last focused item when you navigate away and come back. The problem is it also fires when you are trying to escape downward, which is wrong.

🪤 The Gravity Trap Analogy
RecyclerView has a gravity field that pulls focus back in

Imagine each RecyclerView as a room with a strong gravity field. When you try to exit through the floor (press DOWN), the gravity pulls you back to where you stood before. The room below exists — but you can never reach it unless you override the field.

On STB channel lists, settings rows, and any horizontal TvLazyRow — this trap is always lurking.

Fix 1 — FOCUS_BEFORE_DESCENDANTS (Simplest)

// RecyclerView gives focus to itself first, allowing escape
recyclerView.descendantFocusability = ViewGroup.FOCUS_BEFORE_DESCENDANTS

This tells the ViewGroup: try to focus me before my children. Since RecyclerView itself is focusable, it can "hold" focus briefly and then let FocusFinder route it downward. Works for most settings screens.

Fix 2 — Make RecyclerView Itself Focusable

recyclerView.isFocusable = true
recyclerView.isFocusableInTouchMode = true

Similar effect — when RecyclerView itself can hold focus, the internal gravity-pull to children is broken and focus can escape to the view below.

Fix 3 — Override focusSearch() (Most Powerful)

// Full directional control — best for EPG grids
class MyRecyclerView(ctx: Context) : RecyclerView(ctx) {
    override fun focusSearch(focused: View, direction: Int): View? {
        if (direction == FOCUS_DOWN) {
            return settingsButton  // explicit escape target
        }
        return super.focusSearch(focused, direction)
    }
}

Use Fix 3 when you need precise control over every exit direction. On a STB EPG grid — pressing DOWN from the last channel row should land on a specific control row, not wherever FocusFinder decides. This is how you guarantee it.

Which Fix to Use

ScenarioBest FixWhy
Settings screen, simple listFix 1 — FOCUS_BEFORE_DESCENDANTSOne line, no subclass needed, covers 90% of cases
Multi-row layout, channel gridFix 2 — isFocusable = trueSlightly more explicit, same result, pairs well with focusRestorer
EPG grid, complex 2D navigationFix 3 — override focusSearch()Full control of every exit direction — necessary for complex grids
💡 Focus Memory in RecyclerView
RecyclerView loses "last focused position" during scroll

Views in RecyclerView are recycled as the user scrolls. When a focused view is recycled, focus memory is lost — re-entering the list always focuses item 0. Fix: track focused position manually in your adapter, then call requestFocus() on the correct item when it's bound.

var lastFocusedPosition = 0

override fun onBindViewHolder(holder: VH, position: Int) {
    holder.itemView.setOnFocusChangeListener { _, hasFocus ->
        if (hasFocus) lastFocusedPosition = position
    }
}
RecyclerView traps focus because it overrides onRequestFocusInDescendants to restore its last child — a feature that becomes a bug when you're trying to escape. FOCUS_BEFORE_DESCENDANTS fixes most cases in one line. Override focusSearch() for full directional control on complex STB grids.
Chapter 05

Common STB Focus Bugs

Focus disappears, wrong view wins, highlight invisible — root causes and fixes

Bug 1 — Focus Disappears After Navigation

User presses BACK, or a fragment transaction completes, or a dialog dismisses. D-pad stops working. Nothing responds. No error.

🐛 Root Cause
The focused view was removed from the hierarchy — no re-focus happened

When a View is detached (fragment removed, dialog dismissed, GONE), any focus it held is simply dropped. Android does not automatically move focus to a fallback view. Nothing. Silence.

// Fix: explicitly request focus in onResume
override fun onResume() {
    super.onResume()
    defaultFocusView.post {           // .post{} = wait for layout to complete
        defaultFocusView.requestFocus()
    }
}
⚠️ The .post{} Rule

Always call requestFocus() inside .post {} when the view may not be laid out yet — such as in onResume(), fragment backstack callbacks, or after setVisibility(VISIBLE). Calling it directly before layout silently returns false. The view appears on screen but nothing has focus.

Bug 2 — Focus Jumps to the Wrong View

User presses RIGHT, focus flies to an unexpected view across the screen. FocusFinder's geometry scored something you didn't intend.

🐛 Root Cause
FocusFinder's geometric winner doesn't match user expectation

Often caused by: a partially off-screen view being skipped (next valid candidate is far away), a view positioned at an unexpected Z-angle from the current focus, or a view in a different container scoring lower than the intended target.

// ✅ Fix: explicit wiring
android:nextFocusRight="@id/intendedTarget"

// Or for dynamic content:
override fun focusSearch(focused: View, direction: Int): View? {
    if (direction == FOCUS_RIGHT && focused.id == R.id.channelName) {
        return findViewById(R.id.firstEpgCell)
    }
    return super.focusSearch(focused, direction)
}

Bug 3 — Focus on a Non-Visible View

A view has focus but is scrolled off-screen or set to GONE. FocusFinder only considers visible, on-screen views. Focus appears stuck — no movement in any direction — with no obvious error.

// ❌ Never set focused view to GONE — focus drops into a void
focusedView.visibility = View.GONE

// ✅ Transfer focus first, then hide
nextView.requestFocus()
previousView.visibility = View.GONE

Bug 4 — Focus Highlight Invisible

The correct view has focus. D-pad navigates correctly. But the user has no visual feedback — they can't see which item is selected. On TV this is as bad as broken focus — the product feels unusable.

// Classic View — drawable selector
<selector xmlns:android="...">
    <item android:state_focused="true"  android:drawable="@drawable/btn_focused" />
    <item android:drawable="@drawable/btn_normal" />
</selector>

// Or in onFocusChanged:
override fun onFocusChanged(gainFocus: Boolean, ...) {
    binding.root.strokeWidth = if (gainFocus) 3 else 0
    binding.root.strokeColor = if (gainFocus) Color.WHITE else Color.TRANSPARENT
}

Bug 5 — Overlay Steals Focus (Common STB Bug)

A background service adds an overlay without FLAG_NOT_FOCUSABLE. WMS gives focus to the overlay. D-pad events arrive at the overlay's empty view tree, nobody consumes them, they're dropped. The app behind looks frozen.

// ❌ Visual-only overlay without the right flags
params.flags = WindowManager.LayoutParams.FLAG_NOT_TOUCH_MODAL

// ✅ Visual-only overlay — passes all input through
params.flags = (WindowManager.LayoutParams.FLAG_NOT_FOCUSABLE
        or WindowManager.LayoutParams.FLAG_NOT_TOUCH_MODAL
        or WindowManager.LayoutParams.FLAG_NOT_TOUCHABLE)
5 Anchor Statements · Common STB Bugs
01Focus drops silently when the focused view is removed. No error, no fallback. Always re-request focus in onResume using .post {}.
02Wrong focus jump = FocusFinder geometry mismatch. Fix with explicit nextFocusXxx — it bypasses scoring entirely.
03Never set a focused view to GONE. Transfer focus to another view first. Hiding a focused view sends focus into a void.
04A focus highlight is not optional on TV. Invisible focus = broken product from the user's perspective, even if the logic is perfect.
05Overlay services must set FLAG_NOT_FOCUSABLE on visual-only windows. Missing this flag is the #1 cause of D-pad breaking when an overlay appears.
Most STB focus bugs fall into five patterns: focus dropped after navigation (fix: post + requestFocus), wrong view selected (fix: explicit wiring), hidden focused view (fix: transfer before hiding), invisible highlight (fix: onFocusChanged + drawable), overlay stealing focus (fix: FLAG_NOT_FOCUSABLE). Each is silent — no crash, no log.
Chapter 06

Compose TV Focus

FocusRequester, onFocusChanged, FocusRestorer — and how Compose bridges to Views

How Compose Focus Works Internally

Compose has its own focus system built on FocusRequester and Modifier.focusable(). Internally, Compose hosts all its composables inside an AndroidComposeView — a regular Android View. This means Compose focus ultimately delegates to the same View focus system. The same FocusFinder, the same requestFocus() chain — just wrapped in a Compose API.

The Three Core APIs

1 — FocusRequester: your programmatic focus handle

val focusRequester = remember { FocusRequester() }

Box(
    modifier = Modifier
        .focusRequester(focusRequester)   // attach handle
        .focusable()                       // opt into D-pad navigation
) { ... }

// Auto-focus when screen opens — LaunchedEffect ensures layout is done
LaunchedEffect(Unit) {
    focusRequester.requestFocus()
}
⚠️ Critical Mistake
clickable{} alone does NOT make a composable focusable on TV

On TV, clickable handles SELECT press — but it does not make the composable reachable via D-pad traversal. You must add .focusable() explicitly. Without it, FocusFinder skips this composable entirely — it is invisible to D-pad navigation.

2 — onFocusChanged: react when focus arrives or leaves

Modifier
    .focusable()
    .onFocusChanged { state ->
        if (state.isFocused) {
            borderColor = Color(0xFF1D9E75)  // show teal highlight
            scale = 1.06f
        } else {
            borderColor = Color.Gray
            scale = 1.0f
        }
    }

3 — FocusRestorer: remember last focused item in a list

TvLazyRow(
    modifier = Modifier.focusRestorer()
    // When user navigates away and returns,
    // the last focused item is automatically restored
) {
    items(channels) { channel ->
        ChannelCard(channel)
    }
}

focusRestorer() is the Compose equivalent of overriding onRequestFocusInDescendants() — it remembers which child last had focus and restores it when the container is re-entered. It is a superset of focusGroup().

focusGroup() vs focusRestorer() — What to Use

ModifierWhat It DoesWhen to Use
focusGroup()Groups children for focus traversal. No memory.When you manage focus state yourself (manual focusedRow / focusCol tracking)
focusRestorer()focusGroup() + remembers and restores last focused childWhen you want the framework to handle focus memory automatically
⚡ Don't Mix Both Strategies
Manual focus state + focusRestorer() = unpredictable conflict

If you track focusedRow / focusCol manually and also use focusRestorer(), both will try to restore focus. Whichever fires last wins — and that's non-deterministic. Pick one strategy and own it fully. If you have manual state, use focusGroup(). If you want automatic memory, use focusRestorer() and remove manual tracking.

D-pad Key Interception in Compose

// onPreviewKeyEvent: fires BEFORE default focus traversal
Modifier.onPreviewKeyEvent { keyEvent ->
    if (keyEvent.key == Key.DirectionDown
            && keyEvent.type == KeyEventType.KeyDown) {
        // intercept — prevent FocusFinder from running
        true  // consumed
    } else false
}

// onKeyEvent: fires AFTER default focus traversal
Modifier.onKeyEvent { keyEvent ->
    // handle keys the composable didn't handle itself
    false
}
5 Anchor Statements · Compose TV Focus
01Compose focus is built on AndroidComposeView — it ultimately delegates to the same View focus system as classic Android.
02clickable{} alone is not enough on TV. focusable() must be added explicitly for a composable to participate in D-pad navigation.
03FocusRequester is your programmatic handle. Always use it inside LaunchedEffect to guarantee the composable is composed before requestFocus() fires.
04focusRestorer() is a superset of focusGroup(). Don't mix it with manual focus tracking — pick one strategy per container.
05onPreviewKeyEvent fires before FocusFinder. onKeyEvent fires after. Use preview to intercept, use post to react.
Compose TV focus wraps the View focus system in a cleaner API. FocusRequester replaces requestFocus(), onFocusChanged drives your highlight, focusRestorer() handles list memory. The most dangerous mistake: using clickable without focusable — the view disappears from D-pad navigation silently.
Chapter 07

Production Patterns

Settings screens, EPG grids, overlay focus management — real STB patterns

Pattern 1 — Settings Screen Auto-Focus

Every settings screen on STB needs a default focused item when it opens. Without it, the user lands on the screen and the remote does nothing.

Classic View (Fragment)

override fun onResume() {
    super.onResume()
    // .post{} ensures view is laid out before requestFocus
    firstMenuItem.post { firstMenuItem.requestFocus() }
}

Compose Screen

val firstItemFocus = remember { FocusRequester() }

LaunchedEffect(Unit) {
    // Small delay ensures Compose layout pass is complete
    delay(100)
    firstItemFocus.requestFocus()
}
💡 The delay(100) Pattern

In Compose, LaunchedEffect(Unit) fires immediately after the first composition — but the view may not be fully measured and laid out yet. A short delay(100) (or delay(50)) gives the layout pass time to complete. This is an accepted Compose TV pattern, not a hack.

Pattern 2 — Manual Focus Grid (Your Actual Code)

When you maintain your own focusedRow / focusCol state, the contract is clear: your state IS the focus memory. The framework should not also try to manage it.

// The correct setup — your state drives focus, framework doesn't fight it
Column(modifier.focusGroup()) {                  // focusGroup, NOT focusRestorer
    menuList.forEachIndexed { rowIndex, row ->
        LazyRow(Modifier.focusGroup()) {         // focusGroup, NOT focusRestorer
            itemsIndexed(items = row) { colIndex, tile ->
                val shouldFocus = (focusedRow == rowIndex && colIndex == focusCol)

                // LaunchedEffect drives focus changes reactively
                LaunchedEffect(shouldFocus) {
                    if (shouldFocus) {
                        focusRequesters[rowIndex to colIndex]?.requestFocus()
                    }
                }
            }
        }
    }
}

Pattern 3 — Overlay Focus Management (WindowManager)

Two types of overlays — each needs different flags:

Visual-only overlay (badge, toast, status)

// Passes all input through — underlying app keeps focus
params.flags = (WindowManager.LayoutParams.FLAG_NOT_FOCUSABLE
        or WindowManager.LayoutParams.FLAG_NOT_TOUCH_MODAL
        or WindowManager.LayoutParams.FLAG_NOT_TOUCHABLE)

Interactive overlay (CC panel, PIN dialog)

// Overlay takes focus — underlying app loses D-pad until overlay is dismissed
params.flags = WindowManager.LayoutParams.FLAG_NOT_TOUCH_MODAL
// FLAG_NOT_FOCUSABLE is NOT set — overlay gets focus

// CRITICAL: defocus before removing, so WMS returns token to underlying app
private fun releaseOverlay() {
    composeView?.let { view ->
        val defocusParams = view.layoutParams as? WindowManager.LayoutParams
        defocusParams?.flags = defocusParams?.flags or
                WindowManager.LayoutParams.FLAG_NOT_FOCUSABLE
        wm?.updateViewLayout(view, defocusParams)  // defocus first
    }
    wm?.removeView(composeView)                    // then remove
}
🔍 Why Defocus Before removeView()

When you remove a focusable overlay without first adding FLAG_NOT_FOCUSABLE, WMS drops the focus token without returning it to the underlying window. The underlying app never gets focus back — D-pad stays broken after the overlay closes. Adding the flag first tells WMS: "return this focus token to whoever was focused before." You'll see "Unknown focus tokens, dropping reportFocusChanged" in logcat if you skip this step.

Pattern 4 — EPG Grid 2D Navigation

A 2D grid (rows × columns) is the hardest focus case on STB. FocusFinder often fails — cells are small, tightly packed, and partially off-screen during scroll. The only reliable solution is custom focusSearch().

class EpgGridLayout(ctx: Context) : RecyclerView(ctx) {
    override fun focusSearch(focused: View, direction: Int): View? {
        val position = getChildAdapterPosition(focused)
        val row = position / COLUMNS
        val col = position % COLUMNS

        return when (direction) {
            FOCUS_RIGHT -> if (col < COLUMNS - 1) findViewForAdapterPosition(position + 1) else null
            FOCUS_LEFT  -> if (col > 0) findViewForAdapterPosition(position - 1) else null
            FOCUS_DOWN  -> if (row < ROWS - 1) findViewForAdapterPosition(position + COLUMNS) else controlRow
            FOCUS_UP    -> if (row > 0) findViewForAdapterPosition(position - COLUMNS) else null
            else        -> super.focusSearch(focused, direction)
        }
    }
}
Four production patterns cover most STB focus work: auto-focus on screen open (post + requestFocus), manual grid focus (your state = focus memory, use focusGroup not focusRestorer), overlay management (defocus before removeView), and EPG 2D grids (custom focusSearch — FocusFinder always fails here).
Interview Q&A

Questions & Answers

Tap each question to reveal the answer
How does Android decide which view gets focus when you press RIGHT on a TV remote?+
FocusFinder.findNextFocus() runs. It first checks if the current view has an explicit nextFocusRight set — if yes, that view wins immediately. If not, it collects all focusable, visible views on screen, discards those to the left of the current view's center, and scores the rest using the formula 13 × majorDistance² + minorDistance². Major axis is horizontal distance (for a RIGHT press), minor is vertical offset. Lowest score wins. The 13× multiplier ensures strongly directional alignment wins over diagonal proximity.
What is the difference between focusable and focusableInTouchMode, and why does it matter on STB?+
On touch devices, views need focusableInTouchMode=true to receive programmatic focus in touch mode — this prevents buttons staying visually "selected" after a touch. On STB, isInTouchMode() always returns false because there's no touchscreen. This means the focusableInTouchMode check is bypassed entirely — any view with focusable=true can receive D-pad focus. This is important for shared components used on both mobile and TV: they'll behave differently per platform even with the same flag settings.
Why does a RecyclerView sometimes trap focus and how would you fix it?+
RecyclerView overrides onRequestFocusInDescendants() to restore the last focused child when re-entering the list. This mechanism also fires when trying to escape the list via D-pad DOWN — RecyclerView intercepts the request and refocuses its own child instead of passing focus to the view below. Fixes in order of simplicity: (1) recyclerView.descendantFocusability = FOCUS_BEFORE_DESCENDANTS — one line, works for most cases. (2) Set isFocusable = true on the RecyclerView itself. (3) Override focusSearch() in a custom RecyclerView subclass for full directional control — necessary for EPG grids.
D-pad stops working after an overlay appears on STB. No crash, no error. How do you debug this?+
Classic missing FLAG_NOT_FOCUSABLE bug. When a WindowManager overlay is added without this flag, WMS assigns it focus — it becomes the topmost focusable window. D-pad events arrive at the overlay's view tree. If no view in the overlay consumes them, they are dropped silently. The app behind appears frozen. Debug by running adb shell dumpsys window | grep mCurrentFocus — you'll see the overlay service listed as the focused window instead of your app. Fix: add FLAG_NOT_FOCUSABLE or FLAG_NOT_TOUCH_MODAL or FLAG_NOT_TOUCHABLE for visual-only overlays.
What is the difference between focusGroup() and focusRestorer() in Compose TV?+
focusGroup() groups children for D-pad traversal — focus stays inside the group before escaping. No memory. focusRestorer() does everything focusGroup does, plus it remembers which child last held focus and restores it when the container is re-entered — equivalent to overriding onRequestFocusInDescendants() in a classic ViewGroup. Use focusGroup when you manage focus state manually (your own focusedRow/focusCol tracking). Use focusRestorer when you want the framework to handle focus memory automatically. Never use both — they conflict and produce non-deterministic focus restoration.
Why must you defocus an overlay before calling removeView() on a WindowManager window?+
When you remove a focusable window from WMS without first clearing its focus, WMS drops the focus token. It doesn't know which window should take over — so no window gets focus back. The underlying app behind the overlay permanently loses D-pad input until something else triggers a focus change. The fix: before removeView(), update the overlay's LayoutParams to add FLAG_NOT_FOCUSABLE and call updateViewLayout(). This signals WMS to return the focus token to the window that held it before the overlay appeared. You'll see "Unknown focus tokens, dropping reportFocusChanged" in logcat when this is missing.
Focus disappears after a fragment back-stack pop on STB. How do you fix it?+
The fragment being popped held focus on one of its views. When the fragment is removed, that view is detached and focus is dropped — Android provides no automatic fallback. Fix: in the returning fragment's onResume(), call targetView.post { targetView.requestFocus() }. The .post {} is critical — it defers execution until after the next layout pass, ensuring the view is measured and attached before requestFocus runs. Calling it directly in onResume without post often silently returns false because layout hasn't completed yet.
